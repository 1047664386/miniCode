## 一、Skill 识别与加载系统

### P1: Skill 自动触发识别未实现 ✅

**现状**：已实现基于关键词的 Skill 自动触发机制。

**已实现**：
1. `parseFrontmatter` 升级：支持 `triggers: [keyword1, keyword2]` 数组格式解析
2. `SkillMeta` 扩展：新增 `triggers: string[]` 字段
3. `SkillStore.matchForInput()`：用户输入 → tokenize → 匹配 triggers → 按匹配分排序
4. `renderAutoTriggeredSkills()`：自动触发匹配的 skill 全文注入 context（最多 2 个，每个截断 3000 字）
5. `renderForSystem()` 概览展示 triggers 提示，帮助 LLM 更好判断

**匹配策略**：
- 全串包含匹配：用户输入包含 trigger → 长词(≥3字)权重 2，短词权重 1
- Token 级匹配：输入 token 与 trigger 全词/子串匹配 → 权重 1
- 中文支持：按字 + 二字 bigram 切分

**未实现（P3 远期）**：
- `/skill` slash command 入口
- 超过 30 条 skill 时的 embedding 召回 top-K

### ✅ P2: Skill 热更新 —— 已修复

**修复说明**：`startWatch()` 方法已使用 chokidar 监听 project-level 和 user-level 两个 skills 目录，配置 `ignoreInitial: true, depth: 2`，只监听 `SKILL.md` 文件的 add/change/unlink 事件，触发后自动调用 `this.load()` 全量重新扫描。

**相关代码**：`apps/server/src/skill-store.ts` 第 86-105 行。

### P2: Skill Schema 校验不足

**现状**：frontmatter 解析器是简化的 YAML-lite（只支持 `key: value`），不支持嵌套、数组、多行字符串。

**问题**：如果 skill 定义需要 `triggers: ["deploy", "发布"]` 这样的数组，当前解析器无法处理。

**建议**：引入轻量 YAML 解析（如 `yaml` 库），或至少支持数组语法 `triggers: [a, b]`。

## 二、MCP 集成

### ✅ P1: MCP Tool 参数 Schema 缺失 —— 已修复

**修复方案**：新增 `jsonSchemaToZod()` 函数，将 MCP 的 `inputSchema`（JSON Schema）转换为 zod schema。覆盖 object/string/number/boolean/array/enum/nullable 等常见类型。转换失败时 fallback 到 `z.record(z.string(), z.any())`。

**相关代码**：`apps/server/src/mcp-client.ts` 第 269-396 行

### ✅ P2: MCP 进程生命周期管理粗放

**修复方案**：在 `McpClient` 中新增指数退避自动重连机制。MCP server 异常退出（exit code ≠ 0）时，按 1s → 2s → 4s 延迟最多重连 3 次。`close()` 方法会阻止重连。重连结果通过 `onReconnect` 回调通知 McpManager。

**相关代码**：`apps/server/src/mcp-client.ts`

### P3 → P2: MCP 资源（Resources）协议未实现 — ❌ 跳过

**现状**：只实现了 `tools/list` + `tools/call`，没有实现 `resources/list` + `resources/read`。

**跳过理由**：
- 项目已有 `read_file` + `autoContext` 检索机制覆盖 99% 数据注入场景
- Resources 只在 MCP server 提供非文件结构化数据（如数据库行）时有增量价值，当前项目无此需求
- 实现需要改 McpClient + McpManager + ToolRegistry + 前端展示，改动面大但 ROI 极低
- 面试时如有此问题，可回答"ROI 太低，read_file + 索引检索已覆盖"

## 三、子 Agent 系统

### ✅ P1: 子 Agent 结果推送时机不可控

**修复方案**：在父 turn 的 finally 块中，除了通过 SSE 推送 `subagent_announce` 事件外，还将 announce 消息作为 user message 注入父 session。同时发送 `subagent_followup` SSE 事件通知前端自动触发 follow-up turn，使父 Agent 能在下一轮处理子 Agent 结果。

**相关代码**：`apps/server/src/main.ts` 第 2231-2254 行

### P2: 子 Agent 并发写冲突

**现状**：多个子 Agent 默认共享同一 workspace，并发写文件可能互相覆盖。`worktree` 隔离是可选的，且 Git worktree 创建有开销。

**问题**：
- 两个子 Agent 同时 `write_file` 同一路径 → 后写的覆盖前写的
- 没有 file lock 机制

**建议**：
1. 默认启用 worktree 隔离（至少对写操作）
2. 增加文件级 advisory lock
3. 子 Agent 完成后 merge worktree 结果到主分支

### P2: 子 Agent 流式输出到前端不完整 ✅

**已实现**：
- 统一 `subagentEventBuffer` 替代原 `subagentProgressBuffer`，容纳 text/tool/progress 三类事件
- `child_text` 事件：按 runId 聚合累积，200ms 节流批量 flush，避免逐 token 发送造成带宽风暴
- `child_tool` 事件：频率低，直接写入 buffer
- `child_tool_result` 事件：沿用原逻辑，写入 buffer
- SSE flush 在 ask 和 agent/plan 两个循环中均已接入，每轮事件消费时先 flush 残留 text 累积再逐条发送
- Buffer 容量上限 300 条，超出丢弃最旧事件

**SSE 事件格式**：
- `{ type: 'subagent_text', runId, text, ts }` — 子 Agent 流式文本（200ms 批量）
- `{ type: 'subagent_tool', runId, tool, ts }` — 子 Agent 工具调用
- `{ type: 'subagent_progress', runId, tool, resultPreview, ts }` — 子 Agent 工具结果

## 四、上下文管理

### ✅ P1: Hard Compact 后可能丢失关键推理信息

**修复方案**：在 `summarizeHard()` 中新增决策/推理片段提取。通过中英文决策关键词（"I'll..."、"Let me..."、"我决定..."、"Based on..."等）从 assistant 消息中提取关键推理，保留最多 5 条片段。未匹配到决策关键词时，取 assistant 消息首句作为意图摘要。

**相关代码**：`packages/core/src/context/hard-compact.ts`

### P2: Token 估算精度不足

**现状**：使用启发式公式 `tokens = ceil(cjk/1.35 + other/4) * 1.2`，没有使用真实 tokenizer。

**问题**：
- 对中文内容估算偏保守（多估 20%），导致过早触发压缩
- 对代码片段估算偏差更大（特殊 token 多）

**建议**：
- 接入 `tiktoken`（OpenAI）或 `js-tiktoken` 作为可选注入
- Anthropic 模型使用 Anthropic 自己的 tokenizer 估算

### P3: 没有 Prompt Cache 命中率统计

**现状**：Anthropic Provider 设置了 `cache_control: { type: 'ephemeral' }`，但没有统计缓存命中率和节省的 token 数。

**建议**：在 Anthropic Provider 的 response 中解析 `cache_creation_input_tokens` 和 `cache_read_input_tokens`，上报到 metrics。

## 七、代码检索与索引

### P2: VectorStore 全量内存，大项目 OOM 风险

**现状**：所有向量都存在内存中，JSONL 持久化。万级文件的项目可能有数十万 chunk，向量维度 1536，占用 ~1GB 内存。

**建议**：
1. 增加磁盘索引模式（如 `hnswlib-node` 或 SQLite-vss）
2. 或按文件分片，LRU 淘汰冷 chunk 的向量

### P2: 增量索引的向量更新粒度粗

**现状**：文件变更时，`upsertFile` 会删除该文件所有旧向量，重新 embed 所有 chunk。

**问题**：小改动（改一行注释）也触发全文件重新 embed，浪费 API 调用。

**建议**：做 chunk 级 diff，只重新 embed 变更的 chunk。

### P3: BM25 中文分词简陋

**现状**：中文按单字分词（`[\u4e00-\u9fa5]`），没有词级分词。

**问题**：搜"数据库"时，BM25 会拆成"数""据""库"三个 token，召回噪声大。

**建议**：接入 `jieba` 或 `intl.Segmenter` 做中文分词。

**相关代码**：`apps/server/src/exec-policy.ts`、`apps/server/src/permission-aware-policy.ts`

### P3: SSRF 防护仍有细微缺口

**现状**：`web-fetch-tool.ts` 已实现较完善的 SSRF 防护：

**已实现**：
- ✅ 协议白名单（只允许 http/https，禁止 file:// 等）

**仍可优化（P3 级别）**：

> ⚠️ 定级调整：原 P2 → P3。核心防护（协议限制、DNS 重绑定、重定向限制）已实现。

## 十二、上下文工程与 Token 经济

### P1: System 消息未分层缓存，Prompt Cache 命中率低

**现状**：`buildMessages()` 把 system prompt + autoContext + memory + systemExtras 全部拼在一个 system 消息里。任何一部分变化，整个 system 消息的 Prompt Cache 都会失效。

**问题**：
- Anthropic Prompt Cache 按前缀匹配，system 内容变化 → 全部缓存失效
- 每轮重复发送 ~2000 token 的稳定 system prompt + ~3000 token 的 tools schema

**建议**：
```typescript
// 拆成3段，分层缓存
// 对于支持多 system message 的 provider（OpenAI）：
{ role: 'system', content: systemBase, cacheHint: 'ephemeral' }       // 稳定，100%命中
{ role: 'system', content: rulesAndPermissions, cacheHint: 'ephemeral' } // 半稳定
{ role: 'system', content: dynamicContext }                              // 动态，不缓存
// 对于 Anthropic（只接受单个 system 参数）：
// 在 provider 适配层将多段拼回单条，但按稳定性排序确保稳定前缀不变
```

**注意**：Anthropic API 只支持一个 system 参数。拆分策略需要在 provider 适配层实现——将稳定部分放前面，动态部分放后面，利用 Anthropic 的前缀匹配 cache。

**相关代码**：`packages/core/src/context/builder.ts` 第 174-182 行

### P2: 工具 Schema 每轮全量发送，未做 Progressive Disclosure ✅

**已实现**：基于 AgentMode 的 Progressive Tool Disclosure。

**实现方式**：
- `ToolRegistry.CHAT_ONLY_PROFILE` 已定义（read_file, list_files, grep_search, find_symbol, search_web）
- `ToolRegistry.filter()` 方法创建子 registry（共享同一份 tool 实例）
- 在 `main.ts` 的 runAgent 调用处：
  - `ask` 模式 → 空 registry（无工具）
  - `plan` 模式 → `registry.filter(CHAT_ONLY_PROFILE)`（5 个只读工具，~600 token）
  - `agent` 模式 → 全量 registry（27+ 工具）

**节省**：plan 模式下工具 schema 从 ~3000 token 降至 ~600 token，每轮节省 ~2400 token。

**未实现（P3 远期）**：Agent 模式按任务阶段动态裁剪（探索阶段只给读工具，执行阶段给写工具）

### P2: 缺少 Token 预算分配策略

**现状**：没有显式的 token 预算分配，各路注入硬截断但不统筹。

**建议**：
| 区块 | 预算占比 | 说明 |
|------|---------|------|
| System Prompt | 5% | 固定成本 |
| AutoContext | 0-15% | 按检索质量动态分配 |
| History | 30-50% | 压缩后 |
| User Message | 5-10% | 用户输入 |
| Output | 20% | LLM 生成空间 |

### P3: autoContext 低质量时仍注入

**现状**：无论检索质量如何，autoContext 最多注入 6 条。如果检索结果与用户问题不相关，反而会干扰 LLM 注意力。

**建议**：增加相似度阈值——top-1 相似度 < 0.5 时不注入 autoContext，让 LLM 主动检索。

### P2: acceptAll 两阶段提交依赖 fs.rename 跨设备原子性

**现状**：`acceptAll()` 的 Phase 2 使用 `fs.rename` 将 staging 文件搬到目标路径。`fs.rename` 在同一设备上是原子操作，但如果 `.minicodeide/staging/` 和目标文件不在同一磁盘/分区上，`rename` 会失败（EXDEV 错误）。

**问题**：
- 符号链接场景下，staging 和目标可能在不同的文件系统上
- Docker volume mount 场景下也可能跨设备

**建议**：增加 EXDEV 错误的 fallback——`rename` 失败时改用 `copyFile` + `unlink`。
