# MiniCodeIDE 整体架构与功能概览

> 本文档从宏观视角介绍 MiniCodeIDE 的整体架构、功能模块、技术选型和目录结构。

---

## 一、项目定位

MiniCodeIDE 是一个**自建的 Cursor/Claude-Code 风格 AI 编程 IDE**，核心能力：

1. **AI 对话编程** — 自然语言对话 → Agent 自主读/写/搜索/执行代码
2. **智能代码检索** — BM25 + SymbolGraph + VectorStore 三层混合检索 + RRF 融合
3. **可审查的 AI 改动** — Proposed Edit → 用户审核 → 落盘 → Checkpoint 回滚
4. **多 Provider 动态切换** — 支持 OpenAI/Anthropic/DeepSeek/Ollama 等，运行时切换无需重启
5. **内联 AI 补全** — Ghost Text FIM 风格代码补全
6. **Skill/Rule/MCP 扩展** — 项目级/用户级知识注入 + MCP 工具扩展
7. **子 Agent 调度** — 层级式 Agent 并发执行 + 结果推送

## 四、技术选型与决策

| 维度 | 选型 | 理由 |
|------|------|------|
| 前端框架 | React 18 + Zustand | 生态成熟，Monaco 集成方便，Zustand 轻量 |
| 编辑器 | Monaco + 可选 code-server iframe | Monaco TS 体验满级；VSCode 模式给完整体验 |
| 后端框架 | Express（生产）/ bare-Node（开发） | Express 功能全；bare-Node 启动快 50ms |
| 包管理 | pnpm workspace | Monorepo 标准，hardlink 节省磁盘 |
| 语言 | ESM-only TypeScript | NodeNext 模块解析，`import` 必须 `.js` 后缀 |
| LLM 接入 | 双 Provider（OpenAI 兼容 + Anthropic 原生） | 覆盖主流模型，Anthropic 原生支持 Prompt Cache + Thinking |
| 向量存储 | 内存 + JSONL | 万级 chunk 够用，免部署 Qdrant |
| 代码索引 | BM25 + tree-sitter + VectorStore | 三路召回 + RRF 融合，覆盖关键词/符号/语义三种场景 |
| 会话持久化 | jsonl append-only | 崩溃友好，每 session 一个文件 |
| 云数据库 | Prisma 7 + PostgreSQL | 类型安全，Migration 友好 |
| 沙箱 | E2E（生产）/ Mem（开发） | 云端安全隔离；本地零配置 |
| 打包 | Electron + electron-builder | 跨平台桌面应用 |

## 五、核心功能模块一览

### 5.1 Chat 对话（SSE 流式）

```
用户输入 → Slash预处理 → @rule/@mention解析
  → hybridRetrieve 自动召回上下文
  → Rules激活 → ProjectMemory注入 → Skills概览
  → buildMessages（compaction + 去重）
  → runAgent ReAct循环
    → LLM.chatStream → 流式text/tool_call
    → ToolRegistry.execute → 工具结果
    → SSE推送事件（text/tool_call/tool_result/pending_edit/plan/done）
  → 前端逐字渲染 + 工具调用折叠 + 代码Diff预览
```

### 5.2 代码检索

- **BM25**：关键词精确匹配，毫秒级，免训练
- **SymbolGraph**：tree-sitter AST 抽取，O(1) go-to-def / find-references
- **VectorStore**：语义召回，embedding 后内存余弦相似度
- **RRF 融合**：`score(d) = Σ 1/(k + rank_i(d))`，k=60

### 5.3 AI 代码编辑（Proposed Edit 工作流）

```
LLM调write_file/edit_file
  → proposeEdit审查（checkPermissions）
  → PendingEditStore暂存
  → 前端展示Diff
  → 用户Accept/Reject
  → Accept → 落盘 + Git Checkpoint
  → Reject → 丢弃
```

### 5.4 内联补全（Ghost Text）

```
Monaco trigger character / idle
  → POST /api/complete (prefix + suffix + filename)
  → hybridRetrieve 召回相关代码
  → LLM FIM补全
  → Ghost Text 渲染
```

### 5.5 Skill 系统（Progressive Disclosure）

```
系统启动 → SkillStore.load() 扫描 .minicodeide/skills/ + ~/.minicodeide/skills/
  → renderForSystem() 生成概览（name + description）
  → 注入到 system prompt
  → LLM看到概览 → 判断需要 → 调use_skill(name)
  → SkillStore.loadFull(name) 加载SKILL.md全文 + supportFiles
  → 全文注入到后续消息
```

### 5.6 MCP 集成

```
读取 .minicodeide/mcp.json 配置
  → McpManager.loadAndConnect()
  → 每个server: spawn子进程 → initialize → tools/list
  → registerToolsTo(registry) 注册为 mcp__<server>__<tool>
  → Agent Loop 可调用 MCP 工具
```

### 5.7 子 Agent 调度

```
父Agent调dispatch_subagent({task, role})
  → SubagentManager.spawn()
    → depth/concurrent检查
    → 创建独立子session
    → 构建角色profile + 裁剪工具集
    → 后台 runAgent（独立消息历史）
  → 立即返回 {runId, childSessionId}（不阻塞）
  → 子Agent完成 → deliverAnnounce() → 放入父的pending队列
  → 父下一turn开始 → pickPendingAnnouncements() → 拉取结果
```

## 七、项目亮点

1. **从零构建完整 IDE** — 不是 fork 或插件，从 Agent Loop 到前端 UI 全自研
2. **三路混合检索 + RRF** — 比单一检索方案召回率和准确率都高
3. **四级上下文压缩** — L1→L4 逐层升级，从轻量到紧急，保证长对话不崩
4. **Progressive Disclosure Skill** — 按需加载，节省 context 空间
5. **双后端设计** — Express 全功能 + bare-Node 极速启动
6. **子 Agent Worktree 隔离** — 并发写冲突问题的工程解法
7. **MCP 生态接入** — 一次实现，接入社区 100+ MCP Server

## 八、项目不足与改进方向

1. **Skill 自动触发缺失** — 完全依赖 LLM 自主判断
2. **MCP Tool Schema 缺失** — LLM 无法知道 MCP 工具参数
3. **记忆召回没有上下文关联** — 不结合对话历史和近期活动
4. **向量索引全内存** — 大项目有 OOM 风险
5. **Hard Compact 信息丢失** — 只保留文件路径，丢失推理链
6. **端到端测试缺失** — 只有单元测试
7. **中文分词简陋** — 单字分词，BM25 中文召回差
