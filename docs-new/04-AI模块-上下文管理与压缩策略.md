# AI 核心模块：上下文管理与压缩策略深度分析

> 本文档深入分析 MiniCodeIDE 的上下文构建、四级压缩管线、Token 估算、注入去重等核心机制。

---

## 一、上下文构建全景

### 1.1 buildMessages 组装流程

```
buildMessages(opts) → ChatMessage[]

1. buildSystemPrompt(mode/cwd/os/provider/sandbox/approval)
   → 7段式拼接: identity → tone → conventions → code_refs → tool_discipline → task_flow → safety → [permissions]
   → Plan Mode额外注入5阶段流程

2. 收集context段落：
   ├─ autoContext (hybridRetrieve召回，最多6条，每条截2000字符)
   ├─ explicitContext (@-mentions，每条截3000字符)
   ├─ memory recall (去重via InjectionCache，最多5条，每条截500字符)
   └─ systemExtras (rules/project-memory/skills概览等)

3. compactHistory(history) → 压缩后的历史
   ├─ head: 最早2条（系统指令+首条user query）
   ├─ tail: 从最新向前累加token，至少5条user/assistant
   └─ middle: 有summarizer→LLM摘要; 无→启发式摘要

4. 拼接 [system, ...compactedHistory, user(+_multimodal)]
```

### 1.2 与主流方案对比

| 维度 | MiniCodeIDE | Cursor | Claude Code | ChatGPT |
|------|-------------|--------|-------------|---------|
| System Prompt | 7段式模块化 | 单文件 .cursorrules | CLAUDE.md | 固定 |
| 自动上下文召回 | hybridRetrieve | 代码库索引 | glob+grep | 无 |
| 手动上下文 | @-mentions | @-code/@-doc | 无 | 附件 |
| 压缩策略 | 4级管线 | 未知 | 分层摘要 | 滑动窗口 |
| 注入去重 | InjectionCache | 无 | 无 | 无 |
| 近期活动 | RecentActivityTracker | 无 | 无 | 无 |

**亮点**：
1. **7段式模块化 System Prompt** — 比单文件更灵活，可按 mode/role 动态裁剪
2. **InjectionCache** — 同一 session 不重复注入同一条 memory/chunk，节省 token
3. **RecentActivityTracker** — 解决"那个文件""刚才改的"等指代消解问题

## 二、四级压缩管线

### 2.1 压缩层级总览

```
L0: compactHistory (预防性 token 预算压缩)
    │  触发: 上下文使用 >90% 模型窗口
    │  策略: head/middle/tail 窗口分割 + LLM摘要
    │
L1: snipCompact (消息条数裁剪)
    │  触发: 消息数 >60
    │  策略: 中间塞 [snipped N messages]
    │
L2: microCompact (旧tool_result压缩)
    │  触发: 保留最近3条完整，其余>400字符压缩为一行
    │  策略: 保留工具名+结果摘要
    │
L3: spillLargeToolResults (大结果落盘)
    │  触发: 单条tool_result >4KB
    │  策略: 写入.minicodeide/tool-cache/，messages保留索引行
    │
L4: hardCompact (紧急压缩)
       触发: LLM返回context overflow错误
       策略: 保留stable system + 最后3条，中间全部摘要替代
```

**执行时机**：
- L1+L2+L3：**每轮 LLM 调用前**自动跑一次（`softCompact` 函数）
- L0：在 `buildMessages` 时判断是否需要
- L4：仅在 overflow 错误时触发

### 2.2 L0: compactHistory 详解

```typescript
// Token预算策略
triggerRatio = 0.9    // 上下文用到90%触发
targetRatio = 0.5     // 压缩后保留50%
triggerMaxTokens = 165_000  // 硬上限

// Head/Middle/Tail 分割
head: 最早2条消息（系统指令+首条user query）
tail: 从最新向前累加，至少5条user/assistant，直到超出budget
middle: head与tail之间

// 摘要策略
有summarizer函数 → 调LLM生成摘要（高质量，耗token）
无summarizer → fallbackSummary()（启发式，零token消耗）
```

**启发式摘要提取什么**：
- 文件路径（`/src/foo.ts`）
- 工具使用统计（`read_file x3, edit_file x2`）
- 错误信息（`TypeScript error: ...`）
- 决策关键词（`decided to / chose / because`）

### 2.3 L4: Hard Compact 详解

当 LLM 返回 `prompt_too_long` / `context_overflow` 错误时：

```
1. 保留所有 stable system 消息（带 cachehint: 'ephemeral'）
2. 保留最后3条 user/assistant（含关联的tool消息）
3. 中间全部用启发式摘要替代
4. 摘要消息标注 [HARD-COMPACT]，告知LLM不要试图读取被压缩的tool输出
```

**关键问题**：Hard Compact 丢失了 LLM 的推理链和决策逻辑，只保留了文件路径和工具统计。这可能导致 LLM "忘记" 之前做过什么决策。

### 2.4 与主流压缩方案对比

| 方案 | MiniCodeIDE | MemGPT/Letta | LangChain | Cursor |
|------|-------------|-------------|-----------|--------|
| 压缩层级 | 4级 | 3级(core/summary/archival) | 1级(滑动窗口) | 未知 |
| 摘要方式 | 启发式+LLM混合 | LLM摘要 | 无 | 未知 |
| 大结果处理 | spill落盘 | 全量保留 | 截断 | 截断 |
| 紧急压缩 | Hard Compact | 递归摘要 | 无 | 未知 |
| 压缩后信息量 | 中等（丢失推理链） | 高（LLM摘要保留语义） | 低（直接截断） | 未知 |

**最佳实践**（MemGPT/Letta）：
- Core Memory：当前对话 + 重要摘要，始终在context中
- Summary Memory：历史对话的LLM摘要，按需检索
- Archival Memory：原始对话全量存储，embedding检索召回

**建议**：参考 MemGPT 的三层记忆架构，将 Hard Compact 的摘要改为 LLM 生成结构化摘要（保留决策链），而非启发式提取。

## 三、Token 估算

### 3.1 启发式公式

```typescript
tokens = ceil(cjk/1.35 + other/4) * 1.2
```

- CJK 字符：1 token ≈ 1.35 字符（中日韩文字的 BPE 编码效率低）
- 英文/代码/数字：1 token ≈ 4 字符
- 安全裕度：1.2（多估20%吸收BPE误差）

### 3.2 精度分析

| 内容类型 | 估算误差 | 原因 |
|---------|---------|------|
| 纯英文文本 | +15~20% | BPE对常见词更高效 |
| 中文文本 | +20~30% | CJK的BPE编码更碎片化 |
| 代码 | +10~25% | 特殊token多（缩进/括号/关键字） |
| Markdown | +15~20% | 混合文本和代码 |

### 3.3 与真实 Tokenizer 对比

| Tokenizer | 速度 | 精度 | 依赖 |
|-----------|------|------|------|
| 启发式 | 极快(0ms) | ±20% | 无 |
| tiktoken (OpenAI) | 快(~1ms/KB) | ±2% | WASM ~2MB |
| Anthropic Tokenizer | 中等 | ±2% | API调用 |
| js-tiktoken | 快 | ±2% | JS ~1MB |

**建议**：生产环境注入 `tiktoken`（OpenAI模型）或 Anthropic tokenizer，开发环境保持启发式。

## 四、注入去重缓存 (InjectionCache)

### 4.1 设计

```
sessionId → LRU Cache
  key = normalized URI (e.g., "memory://project/入口是main.ts")
  value = SHA256(content) 前16位
```

**去重逻辑**：
1. 首次注入：记录 key → hash
2. 后续轮次：检查 key 的 hash 是否变化
3. 未变化 → 跳过注入（节省token）
4. 变化 → 更新hash + 重新注入

### 4.2 效果

- 长对话中，重复的 memory/context 不会反复注入
- 典型场景：5条 memory，每轮都检查，如果内容不变则不重复注入，节省 ~2500 token/轮

### 4.3 不足

- 只在 session 级别去重，跨 session 不共享
- hash 变化时（即使只是标点改动），会重新注入全量

## 五、RecentActivityTracker

### 5.1 设计

```
环形缓冲（每session最多30条，TTL 30分钟）
记录: { type: 'edit'|'read'|'search'|'view', path, ts, detail? }
渲染: <recent-activity> system 注入
```

### 5.2 解决的问题

| 用户说 | 需要解析 | Activity帮助 |
|--------|---------|-------------|
| "帮我改一下那个文件" | "那个文件"=? | 最近edit的文件 |
| "刚才那个函数有bug" | "刚才那个"=? | 最近view/search的符号 |
| "搜一下这个关键词" | "这个"=? | 最近的search query |

### 5.3 与主流方案对比

- **Cursor**：无类似机制（依赖代码库索引）
- **Claude Code**：无
- **ChatGPT Canvas**：无

**亮点**：这是一个独创的设计，解决了一个真实的 UX 痛点。

## 六、System Prompt 工程

### 6.1 7段式拼装

```typescript
// 按固定顺序拼接，支持按 mode/role/provider 动态裁剪
[
  identity,       // "你是MiniCodeIDE，一个AI编程助手"
  tone,           // 输出风格："简洁、专业、不过度寒暄"
  conventions,    // 代码规范："ESM imports必须.js结尾"
  code_refs,      // path:line引用格式
  tool_discipline, // 工具使用纪律："read_before_write"
  task_flow,      // 任务流程（含Completion Audit）
  safety,         // 安全边界
  permissions,    // Sandbox x Approval矩阵
]
```

### 6.2 Prompt Cache 优化

Anthropic Provider 在 system + tools 数组末尾标记 `cache_control: { type: 'ephemeral' }`。

```
请求1: system(标记cache) + tools(标记cache) + [history] + user
         → Anthropic缓存 system + tools

请求2: system(标记cache) + tools(标记cache) + [history'] + user'
         → 命中缓存，只算 history + user 的 token
```

**效果**：system prompt ~2000 token，tools schema ~3000 token，每轮节省 ~5000 input token。

### 6.3 Completion Audit（完成审计）

在 `task_flow` 段落中，强制要求 LLM 在声明"完成"之前：

> "把'完成'当作未证明的命题，对每条用户需求逐项找权威证据"
> - 证据不强/间接 → 视为未完成
> - 匹配验证范围和需求范围（不能用窄检查支持宽声明）

**这是对抗幻觉的关键防线**：LLM 经常"自以为做完了"，Completion Audit 强迫它逐项验证。

## 七、改进方向

### 7.1 语义压缩替代启发式压缩

当前 middle 段的摘要要么调 LLM（贵），要么用启发式（丢信息）。建议：

1. 中间方案：用小模型（如 GPT-4o-mini / Claude Haiku）做摘要，成本约为大模型的 1/20
2. 结构化摘要：摘要不只是文件路径，还保留决策链、错误原因、工具结果概要

### 7.2 分层记忆架构

参考 MemGPT 的三层架构：
- **Working Memory**：当前对话 + Recent Activity（始终在context）
- **Summary Memory**：历史对话的LLM摘要（按需检索）
- **Archival Memory**：原始对话全量（embedding检索）

### 7.3 上下文预算分配

当前没有明确的 token 预算分配策略。建议：

| 区块 | 预算占比 | 说明 |
|------|---------|------|
| System Prompt | 5% | 固定成本 |
| 自动召回 | 15% | hybridRetrieve 结果 |
| 历史对话 | 50% | 压缩后的历史 |
| 当前输入 | 10% | 用户消息 + @-mentions |
| 预留 | 20% | 给 LLM 输出空间 |

### 7.4 自适应压缩触发

当前 L1+L2+L3 每轮都跑，即使上下文很空。建议：

- 上下文 <50%：跳过压缩
- 50-80%：只跑 L2+L3
- 80-90%：跑 L1+L2+L3
- >90%：触发 L0

### 7.5 压缩质量评估

增加压缩前后的信息保留率指标：
- 关键实体保留率（文件路径、函数名、错误信息）
- 决策链完整性（LLM 的推理步骤）
- 工具调用可追溯性
