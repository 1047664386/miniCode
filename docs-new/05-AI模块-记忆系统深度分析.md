# AI 核心模块：记忆系统深度分析

> 本文档深入分析 MiniCodeIDE 的记忆系统，包括双 Scope 存储、RRF 混合召回、Auto-Memory 自动抽取，以及与主流记忆方案的对比。

---

## 一、记忆系统架构

### 1.1 双 Scope 存储

```
┌─────────────────────────────────────────────────────┐
│                   MemoryStore                        │
│                                                     │
│  user scope: ~/.minicodeide/user/memories.json      │
│  ├─ 跨项目共享的偏好/经验                            │
│  ├─ "用户喜欢用 TypeScript"                          │
│  ├─ "用户习惯单引号"                                 │
│  └─ "用户偏好 pnpm 而非 npm"                        │
│                                                     │
│  project scope: ~/.minicodeide/projects/<hash>/memories.json │
│  ├─ 项目特有的知识                                   │
│  ├─ "入口是 apps/server/src/main.ts"                │
│  ├─ "Agent Loop 在 packages/core/src/agent/loop.ts" │
│  └─ "这个项目用 ESM-only TypeScript"                │
└─────────────────────────────────────────────────────┘
```

**为什么用双 Scope？**
- user scope：跨项目共享偏好（"我喜欢单引号"），避免每个项目都要重新学
- project scope：项目特有知识（"入口文件在哪"），不同项目互不干扰

### 1.2 存储格式

```typescript
interface MemoryItem {
  id: string;          // 唯一ID
  content: string;     // 记忆内容
  scope: 'user' | 'project';
  category: string;    // 分类：preference/knowledge/convention/...
  importance: number;  // 1-5分，影响召回权重
  tags: string[];      // 标签，辅助检索
  embedding?: number[];// 可选向量，语义检索用
  createdAt: number;
  lastHitAt: number;   // 最后命中时间，用于衰减
  hitCount: number;    // 命中次数
}
```

### 1.3 写入安全

```typescript
// Atomic Write：先写.tmp再rename
async function atomicWrite(filePath: string, data: string) {
  const tmp = filePath + '.tmp';
  await fs.writeFile(tmp, data, 'utf-8');
  await fs.rename(tmp, filePath);  // rename在POSIX上是原子的
}
```

**为什么需要 Atomic Write？**
- 如果进程在写入过程中崩溃，.tmp 文件是不完整的，但原文件完好
- rename 是 POSIX 原子操作，不会出现"写了一半"的状态

## 二、召回算法 v3（双路 RRF 融合）

### 2.1 算法流程

```
recall(query, opts) → MemoryItem[]

1. 词法分支 (BM25-like):
   score = token_tf * idf * importance * decay

2. 语义分支 (cosine similarity):
   score = cosine(query_vec, item_vec) * importance * decay

3. RRF 融合:
   score(d) = Σ 1/(k + rank_i(d))，k=60

4. 排序 + 截断: top-K (默认5)
```

### 2.2 衰减函数

```typescript
// 时间衰减：>30天未命中开始指数衰减
decay(item) {
  const age = (now - item.lastHitAt) / DAY_MS;
  if (age <= 30) return 1.0;
  // 半衰期30天，最低0.1
  return Math.max(0.1, Math.pow(0.5, (age - 30) / 30));
}
```

**设计原理**：
- 最近命中的记忆权重高（"用户刚说喜欢TypeScript"比"3个月前说喜欢JavaScript"更可信）
- 指数衰减而非线性，避免旧记忆突然"消失"
- 最低0.1保底，防止重要但长时间没命中的记忆被完全忽略

### 2.3 Importance 权重

| importance | 说明 | 权重影响 |
|-----------|------|---------|
| 5 | 关键约束（"永远不要删数据库"） | 5x |
| 4 | 重要偏好（"用 TypeScript"） | 4x |
| 3 | 一般知识（"入口是 main.ts"） | 3x |
| 2 | 辅助信息（"测试用 vitest"） | 2x |
| 1 | 低优先级备注 | 1x |

### 2.4 维护机制

```typescript
maintain() {
  // 1. 归档90+天低命中记忆
  // 2. Jaccard去重（相似度>0.8的记忆合并）
  // 3. 控制总量（超过500条时清理低importance+低hitCount的）
}
```

## 三、Auto-Memory 自动记忆抽取

### 3.1 触发条件

```typescript
// 检查是否应该触发自动记忆
shouldTrigger(userMessage: string, turnCount: number): boolean {
  // 1. 消息含偏好信号词（7组正则）
  const preferenceSignals = [
    /以后.*(用|选|写|prefer)/,     // "以后用TypeScript"
    /我喜欢/,                       // "我喜欢单引号"
    /不要.*用/,                     // "不要用var"
    /默认/,                         // "默认用pnpm"
    /总是/,                         // "总是先跑测试"
    /记住/,                         // "记住这个配置"
    /必须/,                         // "必须有类型标注"
  ];

  // 2. 消息长度 >= 20字符
  // 3. 距上次抽取 >= 5 turn
  return hasMatch && userMessage.length >= 20 && (turnCount - lastTriggerTurn) >= 5;
}
```

### 3.2 LLM 抽取流程

```
1. 构造抽取prompt + zod schema
2. callStructured(llm, {
     schema: MemoryExtractSchema,  // { items: [{content, category, scope, importance}] }
     messages: [
       { role: 'system', content: 'Extract memorable facts from the conversation...' },
       { role: 'user', content: recentConversation }
     ]
   })
3. 分类：project_knowledge → project scope; 其他 → user scope
4. 写入 MemoryStore
5. Best-effort：所有错误吞掉，不影响主流程
```

### 3.3 zod Schema 约束

```typescript
const MemoryExtractSchema = z.object({
  items: z.array(z.object({
    content: z.string().max(200).describe('简明事实陈述'),
    category: z.enum(['preference', 'knowledge', 'convention', 'workflow', 'warning']),
    scope: z.enum(['user', 'project']),
    importance: z.number().min(1).max(5),
  })).max(3).describe('最多抽取3条'),
});
```

**亮点**：
- 用 `callStructured` + zod 严格约束输出格式，避免 LLM 生成无效数据
- 限制最多3条，防止一次抽取过多低质量记忆
- Best-effort 原则：记忆是锦上添花，不能影响主流程

## 四、与主流记忆方案对比

### 4.1 对比表

| 维度 | MiniCodeIDE | MemGPT/Letta | LangChain Memory | ChatGPT Memory |
|------|-------------|-------------|------------------|----------------|
| 存储粒度 | Item级 | Message级 | Message级 | Item级 |
| Scope | 双Scope | 单Scope | 单Scope | 全局 |
| 召回方式 | 双路RRF | Embedding检索 | 滑动窗口 | 未知 |
| 自动抽取 | ✅ Auto-Memory | ✅ 自动摘要 | ❌ | ✅ |
| 时间衰减 | ✅ 指数衰减 | ❌ | ❌ | 未知 |
| Importance | ✅ 1-5分 | ❌ | ❌ | 未知 |
| 去重 | ✅ Jaccard | ❌ | ❌ | 未知 |
| 向量检索 | ✅ 可选 | ✅ 必需 | ❌ | ✅ |

### 4.2 亮点

1. **双路 RRF 融合** — 词法+语义双路召回，比单一检索更鲁棒
2. **时间衰减** — 旧记忆权重逐渐降低，避免过时信息干扰
3. **Importance 权重** — 关键约束（如"不要删数据库"）始终高权重
4. **Auto-Memory** — 自动抽取，用户无需手动维护
5. **Atomic Write** — 防崩溃丢数据

### 4.3 不足

1. **触发条件保守** — 7组正则只覆盖有限的偏好表达，很多场景不会触发
2. **无上下文关联召回** — 只看当前消息，不看对话历史和近期活动
3. **无记忆冲突检测** — 新旧记忆矛盾时（"用TypeScript" vs "用JavaScript"），没有解决
4. **embedding 可选** — 没有 embedding 时退化为纯词法，语义召回能力差
5. **没有记忆编辑/删除 UI** — 用户无法修正错误的记忆

## 五、深入讨论：记忆系统的核心挑战

### 5.1 记忆一致性

**问题**：用户在不同时间说了矛盾的话
- 1月："我喜欢用 JavaScript"
- 3月："我以后都用 TypeScript"

**当前处理**：两条记忆都存在，召回时 importance + recency 决定哪条权重更高。

**最佳实践**：
- 检测到与已有记忆矛盾 → 用 LLM 判断是否应该覆盖
- 记录记忆的"来源对话"和"创建时间"，方便溯源

### 5.2 记忆污染

**问题**：LLM 可能抽取了错误的"事实"
- 用户开玩笑说"我喜欢删库"
- LLM 严肃地记为 preference

**当前处理**：Importance 分数可以降低低置信度记忆的权重，但没有自动检测"玩笑 vs 认真"。

**最佳实践**：
- 抽取时增加置信度字段
- 低置信度的记忆权重低 + 不主动召回
- 增加用户确认机制（"我注意到你提到...，要记住吗？"）

### 5.3 记忆召回的精准度

**问题**：5条记忆全部注入，可能包含无关内容
- 用户问"帮我修这个bug"，注入了"用户喜欢单引号"这条无关偏好

**当前处理**：基于 query 的 RRF 召回，但 query 可能很短（"修bug"），召回不精准。

**最佳实践**：
- 召回 query = 用户消息 + 对话上下文 + 近期活动的融合
- 增加"指代消解"：先解析"那个函数"指的是什么，再召回相关记忆
- 召回后做 LLM rerank，过滤掉真正无关的

## 六、改进方向

### 6.1 短期（1-2周）

1. **放宽触发条件** — 增加 project_knowledge 信号词
2. **记忆召回增强** — query = 用户消息 + RecentActivity + 对话摘要
3. **记忆冲突检测** — 新记忆与旧记忆 Jaccard > 0.8 时，用 LLM 判断是否覆盖

### 6.2 中期（1-2月）

4. **结构化记忆** — 不同 category 用不同 schema（preference用key-value，knowledge用三元组）
5. **记忆编辑 UI** — 前端增加记忆管理面板
6. **记忆溯源** — 每条记忆记录来源对话ID，方便回溯

### 6.3 长期（3+月）

7. **跨项目知识图谱** — 将记忆组织为知识图谱，支持推理
8. **主动记忆** — LLM 在工作中发现的重要事实主动记忆（不依赖用户表达）
9. **记忆蒸馏** — 定期用大模型对旧记忆做摘要/合并，保持记忆库精炼
