# AI 核心模块：代码检索与知识库深度分析

> 本文档深入分析 MiniCodeIDE 的三层混合检索系统：BM25、SymbolGraph、VectorStore，以及 RRF 融合、Symbol-Aware 切片等核心机制。

---

## 一、三层索引架构

### 1.1 为什么需要三层？

| 检索需求 | 最佳层 | 示例 |
|---------|--------|------|
| 精确符号名查找 | BM25 + SymbolGraph | "buildIndex 在哪" |
| 自然语言意图搜索 | VectorStore | "哪里处理鉴权" |
| 代码导航（定义/引用） | SymbolGraph | "谁调用了 runAgent" |
| 混合查询 | RRF 融合 | "修改 Agent Loop 的错误恢复逻辑" |

**核心洞察**：没有任何一种检索能覆盖所有场景。

- 纯向量召回对精确符号名很差：`buildIndex` 这种 token 反而被语义稀释
- 纯 BM25 对自然语言意图差："哪里处理鉴权" → BM25 可能匹配到注释里的"鉴权"
- SymbolGraph 对导航类查询无与伦比：O(1) 的 go-to-def / find-references

### 1.2 三层协作流程

```
用户查询 "修改 Agent Loop 的错误恢复逻辑"
  │
  ├─ BM25: 搜索 "Agent" "Loop" "error" "recovery"
  │  → 命中: loop.ts, error-recovery.ts, hooks.ts
  │
  ├─ SymbolGraph: fuzzyFind("Agent Loop") + findReferences("runAgent")
  │  → 命中: loop.ts:runAgent, main.ts:调用runAgent
  │
  ├─ VectorStore: embed("修改 Agent Loop 的错误恢复逻辑") → 余弦相似度
  │  → 命中: loop.ts中error-recovery相关chunk, error-recovery.ts
  │
  └─ RRF融合: score(d) = Σ 1/(k + rank_i(d))
     → 最终排序: loop.ts, error-recovery.ts, hooks.ts, main.ts
```

## 二、BM25Index 详解

### 2.1 算法

标准 BM25 公式：

```
score(D, Q) = Σ IDF(qi) × (f(qi, D) × (k1 + 1)) / (f(qi, D) + k1 × (1 - b + b × |D| / avgdl))
```

参数：k1=1.5, b=0.75（Elasticsearch 默认值）

- k1 控制词频饱和度：词出现次数越多分数越高，但增速递减
- b 控制文档长度归一化：b=1 完全归一化，b=0 不归一化

### 2.2 分词策略

```typescript
// 中英混合分词
tokenize(text) {
  // 1. 提取CJK单字
  const cjk = text.match(/[\u4e00-\u9fa5]/g) ?? [];
  // 2. 提取英文/代码token（camelCase/snake_case拆分）
  const eng = text.match(/[a-zA-Z][a-z0-9]+|[A-Z][A-Z0-9]+/g) ?? [];
  // 3. 停用词过滤
  return [...cjk, ...eng].filter(t => !STOP_WORDS.has(t.toLowerCase()));
}
```

**中文分词问题**：
- 当前按单字分词："数据库" → "数" "据" "库"
- 搜"数据库"时，BM25 会匹配任何包含这三个字的文档，噪声很大
- **改进**：接入 `jieba` 或 `intl.Segmenter` 做词级分词

**代码分词亮点**：
- camelCase 拆分：`buildIndex` → `build` `Index`
- snake_case 拆分：`run_agent` → `run` `agent`
- 这让 BM25 能匹配部分名称

### 2.3 增量更新

```typescript
// 文件变更时
removeByPath(filePath);  // 删除旧倒排索引
upsertFile(filePath, chunks);  // 重建该文件的索引
```

**性能**：O(M) 其中 M = 变更文件的 chunk 数，不影响其他文件的索引。

## 三、SymbolGraph 详解

### 3.1 数据结构

```typescript
class SymbolGraph {
  // name → SymbolInfo[]（同名符号查找）
  byName: Map<string, SymbolInfo[]>;

  // path → FileFacts（文件级符号/imports/calls）
  byPath: Map<string, FileFacts>;

  // callee name → Reference[]（被谁调用了）
  callIndex: Map<string, Reference[]>;

  // importedName → ImportInfo[]（谁import了它）
  importIndex: Map<string, ImportInfo[]>;
}
```

### 3.2 tree-sitter AST 抽取

```
源代码 → tree-sitter parse → AST
  ├─ 提取函数/类/接口/类型定义 → byName
  ├─ 提取 import 语句 → importIndex
  ├─ 提取函数调用 → callIndex
  └─ 整合为 FileFacts → byPath
```

**支持的语言**：TypeScript/JavaScript（可扩展其他 tree-sitter 语法）

### 3.3 查询能力

| 查询 | 方法 | 复杂度 |
|------|------|--------|
| 精确查找符号 | `byName.get(name)` | O(1) |
| 模糊查找符号 | `fuzzyFind(query)` | O(N) 精确>前缀>包含>子序列 |
| 查找引用 | `findReferences(name)` | O(1) callIndex + importIndex |
| 文件大纲 | `symbolsInFile(path)` | O(1) byPath |
| 全量文件列表 | `allFiles()` | O(N) byPath keys |

### 3.4 与 LSP 的关系

| 维度 | SymbolGraph | LSP |
|------|------------|-----|
| 构建时机 | 索引时（离线） | 打开文件时（在线） |
| 响应速度 | <1ms（内存Map） | 10-100ms（进程通信） |
| 准确度 | 静态分析 | 增量类型检查 |
| 覆盖范围 | 全项目 | 已打开的文件 |
| 依赖 | tree-sitter | typescript-language-server |

**互补设计**：SymbolGraph 做快速粗检索，LSP 做精确导航。AI Agent 优先用 SymbolGraph 做搜索，前端用户操作用 LSP 做跳转。

## 四、VectorStore 详解

### 4.1 数据结构

```typescript
class VectorStore {
  items: VectorItem[];  // 全量内存
  // { id, path, startLine, endLine, text, vector: number[] }
}

// 持久化: JSONL 格式（每行一个item，便于增量append）
// .minicodeide/vectors.<embedder-name>.jsonl
```

### 4.2 搜索算法

```
search(queryVector, topK):
  1. 对所有items计算余弦相似度（= L2归一化后的点积）
  2. 排序取topK
```

**复杂度**：O(N × D)，N=向量数，D=维度（1536 for OpenAI）

**优化方向**：
- 小项目（<10K chunks）：暴力搜索足够
- 大项目：引入 HNSW 索引（`hnswlib-node`，~1ms/topK）

### 4.3 Embedder 策略

```typescript
createEmbedder(config):
  if (config.provider === 'openai' || config.provider === 'ollama')
    → OpenAIEmbedder (API调用)
  else
    → HashEmbedder (FNV-1a hashing trick, 256维, 零依赖)
```

**HashEmbedder 兜底**：
- 无 API Key 时自动降级
- 使用 FNV-1a 哈希将文本映射到 256 维向量
- 质量远不如真实 embedding，但保证"有总比没有强"
- 适合开发和测试场景

## 五、Symbol-Aware 切片

### 5.1 核心思想

传统 naive 切片（按行数切块）的问题：
- 一个函数可能被切到两个 chunk，上下文断裂
- LLM 拿到半个函数，无法理解完整逻辑

Symbol-Aware 切片：
- 每个 top-level symbol 一个 chunk
- chunk 包含上方紧邻的注释（≤8行）
- 长 chunk（>200行）二级切分（header + body）
- 文件 preamble（imports + 顶部注释）单独成 chunk
- gap（散文档）也收集

### 5.2 切片流程

```
源文件 → tree-sitter 提取symbols → 排序去重
  │
  ├─ Preamble chunk: 第1行 → 第一个symbol之前
  │
  ├─ Symbol chunks: 逐个symbol切片
  │  ├─ 短symbol（≤200行）→ 单chunk
  │  └─ 长symbol（>200行）→ header(20行) + body(80行/chunk)
  │
  ├─ Gap chunks: symbol之间散落的内容
  │
  └─ Tail chunk: 最后一个symbol之后的内容
```

### 5.3 收益

| 指标 | Naive 切片 | Symbol-Aware |
|------|-----------|-------------|
| Top-1 召回准确率 | ~60% | ~80% |
| 上下文完整性 | 低（经常截断函数） | 高（函数完整） |
| chunk 数量 | 多（每40行一个） | 少（每函数一个） |
| 嵌入成本 | 高（多chunk多API调用） | 低（chunk少） |

## 六、RRF 融合搜索

### 6.1 算法

```
hybridRetrieve(index, embedder, query, topK, reranker):

1. BM25.search(query, overFetch=topK*3)
2. VectorStore.search(embed(query), overFetch=topK*3)

3. RRF融合:
   score(d) = Σ 1/(k + rank_i(d))
   k=60 (标准参数，平衡top-1和长尾)

4. 可选 cross-encoder rerank:
   对 topK*3 候选重排 → 取 topK

5. 返回 topK 结果
```

### 6.2 为什么用 RRF 而不是加权求和？

| 方法 | 优点 | 缺点 |
|------|------|------|
| 加权求和 | 简单 | 需要归一化不同尺度的分数 |
| RRF | 无需归一化、对异常值鲁棒 | 丢失绝对分数信息 |
| CombMNZ | 考虑多系统重合 | 对单系统命中的结果不友好 |

RRF 的关键优势：**只需要排名，不需要分数**。这避免了一个大问题——BM25 分数和余弦相似度的尺度完全不同，直接加权没有意义。

### 6.3 overFetch 策略

```
overFetch = topK * 3
```

每路多召回3倍候选，原因：
- 单路可能漏掉一些结果（BM25可能漏掉语义相关的，Vector可能漏掉关键词匹配的）
- 3倍是经验值，平衡召回率和计算成本
- RRF融合后再取 topK，保证最终结果质量

### 6.4 Reranker

可选的 cross-encoder rerank：
- 输入：(query, candidate) 对
- 输出：相关性分数
- 模型：`cross-encoder/ms-marco-MiniLM-L-6-v2`（本地）或 API
- 只对 topK*3 候选做 rerank，避免全量计算

## 七、索引增量更新

### 7.1 IndexWatcher

```typescript
// chokidar 监听工作区文件变更
chokidar.watch(workspace, {
  ignored: /node_modules|\.git|\.minicodeide/,
  ignoreInitial: true,  // 忽略初始扫描（已在buildIndex处理）
})
.on('change', (path) => upsertFile(path))
.on('add', (path) => upsertFile(path))
.on('unlink', (path) => removeFile(path));
```

### 7.2 三层索引的增量策略

| 索引 | 增量方式 | 开销 |
|------|---------|------|
| BM25 | removeByPath + upsertFile | 极低 |
| SymbolGraph | upsert(facts) | 低（tree-sitter解析单文件） |
| VectorStore | upsertFile → 重新embed变更文件所有chunk | 中（API调用） |

**VectorStore 增量的问题**：
- 小改动（改一行注释）也触发全文件重新 embed
- 一个文件可能有10+个chunk，每个都要调embedding API

**改进方向**：chunk 级 diff，只重新 embed 变更的 chunk。

## 八、与主流检索方案对比

| 维度 | MiniCodeIDE | Cursor | Sourcegraph | GitHub Copilot |
|------|-------------|--------|------------|----------------|
| 索引方式 | 本地三层 | 自研索引 | 全量代码搜索 | 服务器端索引 |
| 词法检索 | BM25 | 未知 | Zoekt (trigram) | 未知 |
| 语义检索 | VectorStore | 向量检索 | 无 | 向量检索 |
| 符号检索 | tree-sitter | LSP | SCIP | 未知 |
| 融合策略 | RRF | 未知 | 无 | 未知 |
| 增量更新 | chokidar | 文件监听 | 定期reindex | 服务器push |
| 切片策略 | Symbol-Aware | 未知 | 文件级 | 行级 |

**亮点**：
1. 三层检索 + RRF 融合是完整的混合检索方案
2. Symbol-Aware 切片提升召回准确率 20-30%
3. 全本地化，无需部署外部服务

**不足**：
1. 向量索引全内存，大项目有 OOM 风险
2. 中文分词简陋，BM25 中文召回差
3. 没有 reindex 调度（全量重建是手动触发）
