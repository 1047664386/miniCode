# 数据库设计、权限系统与打包构建

> 本文档分析 MiniCodeIDE 的数据库设计（Prisma + PostgreSQL）、权限与安全体系、Electron 打包与构建流程。

---

## 一、数据库设计

### 1.1 双模式存储

MiniCodeIDE 有两套存储实现：

| 模式 | 实现 | 适用场景 |
|------|------|---------|
| 本地模式 | jsonl append-only（每session一个文件） | 桌面版/开发 |
| 云端模式 | Prisma 7 + PostgreSQL | 云服务/多租户 |

### 1.2 Prisma Schema（云端）

```prisma
model User {
  id            String   @id @default(cuid())
  email         String?  @unique
  username      String?  @unique
  password      String?  // bcrypt hash
  wxOpenId      String?  @unique  // 微信登录
  llmKeyEnc     String?  // BYOK 加密存储
  usedTokens    Int      @default(0)
  freeQuota     Int      @default(50000)  // 免费token额度
  sessions      Session[]
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
}

model Session {
  id            String    @id @default(cuid())
  userId        String
  user          User      @relation(fields: [userId], references: [id])
  title         String    @default("New Chat")
  mode          String    @default("work")  // work | code
  workspaceRoot String?
  remoteUser    String?   // 微信遥控用户
  interruptedTurn Json?   // 崩溃恢复用
  messages      Message[]
  turns         Turn[]
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  @@index([userId, updatedAt(sort: Desc)])
  @@index([remoteUser])
}

model Message {
  id            String  @id @default(cuid())
  sessionId     String
  session       Session @relation(fields: [sessionId], references: [id])
  role          String  // system | user | assistant | tool
  content       String  // TEXT
  ts            BigInt
  toolName      String? // 工具名（role=tool时）
  pendingEditId String? // 关联的PendingEdit

  @@index([sessionId, ts])
}

model Turn {
  id            String  @id @default(cuid())
  sessionId     String
  session       Session @relation(fields: [sessionId], references: [id])
  userMessage   String  // 用户输入
  status        String  @default("pending") // pending | done | interrupted
  chunks        Json    // 流式事件数组
  tools         Json    // 工具调用记录数组
  createdAt     DateTime @default(now())
}
```

### 1.3 关键索引分析

| 索引 | 用途 | 查询场景 |
|------|------|---------|
| `userId + updatedAt` | 按用户查最近会话 | 会话列表页 |
| `sessionId + ts` | 按会话查消息时序 | 消息加载/恢复 |
| `remoteUser` | 微信遥控查找 | 微信消息路由 |
| `email UNIQUE` | 登录 | 用户认证 |
| `wxOpenId UNIQUE` | 微信登录 | 微信OAuth |

### 1.4 本地 jsonl 存储设计

```typescript
// 每个 session 一个 .jsonl 文件
// 路径: ~/.minicodeide/sessions/<sessionId>.jsonl

// 事件类型：

// 崩溃恢复：

```

### 1.5 与主流方案对比

| 维度 | MiniCodeIDE | Cursor | ChatGPT | Vercel v0 |
|------|-------------|--------|---------|-----------|
| 本地存储 | jsonl | SQLite | 无 | 无 |
| 云端存储 | PostgreSQL | 未知 | PostgreSQL | PostgreSQL |
| 崩溃恢复 | interruptedTurn | 未知 | 无 | 无 |
| 流式持久化 | chunk事件 | 未知 | 无 | 无 |
| Token计费 | usedTokens | 未知 | ✅ | ✅ |
| BYOK加密 | ✅ llmKeyEnc | ✅ | N/A | N/A |

**亮点**：jsonl append-only 是最简单最可靠的本地存储方案，崩溃友好。

**不足**：
1. Token 计费没有区分 input/output/cache
2. 没有 Message 版本控制（编辑消息后旧版本丢失）
3. 没有数据导出/导入功能

## 二、权限与安全体系

### 2.1 Sandbox x Approval 矩阵

```
              never    unless_trusted   on_failure   on_request   granular
read_only      ✗         ✗               ✗            ✗            ✗
workspace_w    ✓         ✓               ✓            ✓            ✓
danger_full    ✓         ✓               ✓            ✓            ✓

✗ = 该操作在对应sandbox下禁止
✓ = 需要对应级别的approval

read_only: 只读（不能写文件、不能执行命令）
workspace_w: 可写工作区（写文件、执行命令）
danger_full: 完全访问（任何操作）
```

### 2.2 Approval 流程

```
工具调用请求
  │
  ├─ checkPermissions(tool, sandbox, approval)
  │  ├─ sandbox=read_only && tool=write_file → BLOCK
  │  ├─ approval=never → AUTO-APPROVE
  │  ├─ approval=on_request && requiresApproval=true → 弹窗确认
  │  └─ approval=granular → 按工具粒度确认
  │
  ├─ 用户确认 → 执行
  └─ 用户拒绝 → 返回拒绝信息给LLM
```

### 2.3 安全防护层级

| 层级 | 防护 | 实现 |
|------|------|------|
| L1: 路径安全 | 文件操作防逃逸 | `isPathSafe(base, target)` 检查路径是否在workspace内 |
| L2: 命令安全 | 危险命令检测 | `execPolicy` 三级判定（allow/deny/ask） |
| L3: 网络安全 | SSRF防护 | web_fetch禁止内网IP |
| L4: MCP安全 | 命令白名单 | `mcp-allowlist.json` 限制可spawn的命令 |
| L5: 认证安全 | API Token | `timingSafeEqual` 防时序攻击 |
| L6: 数据安全 | Atomic Write | 记忆文件先写.tmp再rename |

### 2.4 API 认证

```typescript
// 双Token认证
authMiddleware(req, res, next) {
  // 1. Bearer Token（header）
  const bearer = req.headers.authorization?.replace('Bearer ', '');
  // 2. X-Mini-Token（自定义header）
  const miniToken = req.headers['x-mini-token'];

  const token = bearer || miniToken;
  if (!token) return res.status(401).json({ error: 'Unauthorized' });

  // timingSafeEqual 防时序攻击
  const expected = Buffer.from(CONFIG_TOKEN);
  const actual = Buffer.from(token);
  if (expected.length !== actual.length) return res.status(401).end();
  if (!crypto.timingSafeEqual(expected, actual)) return res.status(401).end();

  next();
}
```

### 2.5 云端认证（JWT）

```typescript
// server-cloud JWT认证
authPlugin(fastify, opts) {
  fastify.addHook('onRequest', async (req, reply) => {
    // 白名单路由免鉴权
    if (isWhitelisted(req.url)) return;

    // 从cookie或Authorization header获取JWT
    const token = req.cookies['mci_jwt'] || extractBearerToken(req);

    // 验证JWT
    const payload = jwt.verify(token, JWT_SECRET);
    req.userId = payload.sub;
  });
}
```

|------|-------------|--------|-------------|--------|

## 三、打包与构建

### 3.1 Electron 打包配置

```javascript
// electron-builder 配置
{
  appId: 'com.minicodeide.app',
  productName: 'MiniCodeIDE',

  // 额外资源
  extraResources: [
    { from: 'dist/server', to: 'server' },         // Express后端
    { from: 'dist/desktop', to: 'renderer' },       // React前端
    { from: 'resources/vscode-bridge-ext', to: 'vscode-bridge-ext' },
    { from: 'resources/code-server', to: 'code-server', filter: ['**/*'] },
  ],

  // asar打包（但排除native modules）
  asar: true,
  asarUnpack: [
    '**/sherpa-onnx/**',  // 语音识别native模块
  ],

  // 平台目标
  mac: {
    target: [
      { target: 'dmg', arch: ['x64', 'arm64'] },
      { target: 'zip', arch: ['x64', 'arm64'] },
    ],
    category: 'public.app-category.developer-tools',
  },
  win: {
    target: [{ target: 'nsis', arch: ['x64'] }],
  },
  linux: {
    target: [
      { target: 'AppImage', arch: ['x64'] },
      { target: 'deb', arch: ['x64'] },
    ],
  },

  // 自动更新
  publish: {
    provider: 'github',
    owner: '...',
    repo: '...',
  },
}
```

### 3.2 构建流程

```
1. pnpm -r build
   ├─ packages/core → tsc
   ├─ packages/indexer → tsc
   ├─ apps/server → esbuild bundle
   ├─ apps/server-node → esbuild bundle
   └─ apps/desktop → vite build

2. stage-code-server.mjs
   ├─ 检测host架构
   ├─ 下载对应平台code-server (~80MB)
   ├─ 多镜像降级（GitHub → Mirror1 → Mirror2）
   └─ 缓存到 .cache/

3. electron-builder
   ├─ 打包server + renderer + code-server
   ├─ 生成dmg/zip(Mac), nsis(Win), AppImage/deb(Linux)
   └─ 上传到GitHub Releases

4. 自动更新
   ├─ 启动30s后首次检查
   ├─ 每4小时检查
   └─ electron-updater后台下载 + 下次启动安装
```


## 四、改进方向

### 4.1 数据库

1. **Token 精细计费**：区分 input/output/cache token，按模型计费
2. **Message 版本控制**：编辑消息时保留旧版本
3. **数据导出**：支持导出为 Markdown / JSON
4. **会话搜索**：增加 full-text search 支持搜索历史会话

### 4.2 权限

1. **Token 过期机制**：JWT 或 TOTP 动态 token
2. **SSRF 增强**：DNS rebinding 防御 + 重定向限制
3. **操作审计日志**：记录所有工具调用和文件操作
4. **RBAC**：云端多租户角色权限控制

### 4.3 构建

1. **Win/Linux 代码签名**：避免SmartScreen/安全警告
2. **增量更新**：只下载变更部分（bsdiff）
3. **code-server 按需下载**：首次使用时再下载
4. **CI/CD**：GitHub Actions 自动构建 + 发布
