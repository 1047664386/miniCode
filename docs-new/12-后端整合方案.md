> 日期：2026-06-10 | 版本：v1.0 | 状态：待评审

## 一、现状诊断

### 1.1 三个后端并存

| 后端 | 技术栈 | 代码量 | 用途 | 端口 |
|------|--------|--------|------|------|
| `apps/server` | Express | 2488 行（单文件 main.ts） | Electron 生产构建 | 5174 |
| `apps/server-node` | bare Node | 1310 行（register.ts）+ 共享模块 | Vite 开发模式 | 5175 |
| `apps/server-cloud` | Fastify + Prisma + JWT | 完全独立 | Web 云端 | 4000 |

### 1.2 三个前端消费端

| 消费端 | 入口 | 使用的后端 | Store |
|--------|------|-----------|-------|
| **Electron IDE** | `<App/>` via `main.tsx` | server(生产) / server-node(开发) | `useStore` |
| **Agent Window** | `<AgentsApp/>` via `?window=agents` | server-node(桌面) + server-cloud(已登录时) | `useAgentsStore` |
| **Cloud Web** | `<AgentsApp/>` via `apps/web` | server-cloud | `useAgentsStore` |

### 1.3 核心矛盾

```
                    ┌──────────────────────────────────────────────┐
                    │            apps/desktop (React)              │
                    └──┬────────────┬────────────┬────────────────┘
                       │            │            │
          /api (dev)   │  /cloud-api│  /api (electron)
          → :5175      │  → :4000   │  → :5174
                       │            │
    ┌──────────────────▼──┐    ┌────▼─────────────────┐
    │  server-node        │    │  server (Express)     │
    │  bare Node 1310行   │    │  2488行巨型单文件      │
    │  Services 容器 ✅    │    │  全局变量 + 闭包 ❌    │
    │  workspace 热切换 ✅ │    │  无热切换 ❌           │
    │  openSse 封装 ✅    │    │  内联 SSE ❌           │
    └──────┬──────────────┘    └──────┬─────────────────┘
           │ 16+模块相对路径导入 ─────►│
           │ (业务层共享, HTTP层重复)    │
           └─────────────────────────────┘

    ┌──────────────────────────────────────────────────────┐
    │  server-cloud (Fastify + Prisma + JWT)               │
    │  完全独立，不共享代码，SSE 协议不同                     │
    └──────────────────────────────────────────────────────┘
```

| 问题 | 严重度 | 说明 |
|------|--------|------|
| server 和 server-node 路由层 ~80% 重复 | 🔴 | 修 bug 要改两处，已有 3 次遗漏先例 |
| Electron 用 server，dev 用 server-node | 🔴 | 两套代码两条路线，SSE 通道、chat handler 行为不一致 |
| Chat handler 功能深度差距 | 🔴 | server-node 缺少 plan 模式、hooks、session 锁、checkpoint 等 14 项关键特性 |
| 2 个持久 SSE 通道职责不清 | 🟡 | `/api/fs/events` + `/api/composer/events`，前端需 2 个 EventSource |
| SSE 协议本地 vs 云端不统一 | 🟡 | 本地 `data: {type:...}` vs 云端 `event: text\ndata: {delta:...}` |
| server 2488 行巨型单文件 | 🟡 | 无法维护，无法测试 |
| 3 个函数复制粘贴 | 🟢 | `detectMultiStepHint`、`globToRegex`、`runDiagnostics` |

## 二、方案选型

### 2.1 框架对比

| 维度 | Express (现状 server) | bare Node (现状 server-node) | NestJS |
|------|---------|-----|--------|
| 启动速度 | ~500ms | **<50ms** | ~1s+ |
| 依赖体积 | ~600KB | **0** | ~5MB |
| 维护性 | 中（但写成单文件就差） | 中（需自律） | **最好**（强约束） |
| SSE 支持 | 原生 res.write | **封装 openSse** | 需适配 |
| 适合规模 | 中小型 | 中小型 | 中大型 |

### 2.2 结论

**选择 bare Node（在 server-node 基础上统一）**，理由：

1. **启动最快** — Electron 内嵌子进程，冷启动 <50ms 是硬指标
2. **零框架依赖** — 无版本锁定、安全漏洞风险
3. **架构更优** — 已有 Services 容器、openSse 封装、workspace 热切换
4. **NestJS 过度设计** — 50 个端点的本地 IDE 后端不需要装饰器 + reflect-metadata + 模块系统
5. **已有可工作的实现** — server-node 已经跑了，路线风险最低

**server-cloud 不动** — 它是完全独立的系统（JWT + PostgreSQL + Sandbox），改动它不在本次范围内。

## 三、API 缺口详细分析

### 3.1 server 独有但 server-node 缺失的路由

| 路由 | 前端是否使用 | 风险 |
|------|-------------|------|
| `GET /api/subagents/profiles` | **是** — SubagentLauncher.tsx | 🔴 删除会断裂 |
| `POST /api/subagents/spawn` | **是** — SubagentLauncher.tsx | 🔴 删除会断裂 |
| `GET /api/hooks/metrics` | 否 | 🟢 无影响 |
| `GET /api/bg/list` | 否（前端无调用） | 🟢 无影响 |

### 3.2 server-node 独有但 server 缺失的路由

| 路由 | 前端是否使用 | 说明 |
|------|-------------|------|
| `DELETE /api/file` | **是** — FileTree | Express 版缺失 |
| `POST /api/file/rename` | **是** — FileTree | Express 版缺失 |
| `POST /api/folder` | **是** — FileTree | Express 版缺失 |
| `POST /api/file/reveal` | **是** — FileTree | Express 版缺失 |
| `GET /api/fs/list-abs` | **是** — FolderPickerModal | Express 版缺失 |
| `GET /api/workspace` | **是** — agents-window | Express 版缺失 |
| `POST /api/workspace/switch` | **是** — FileTree, WelcomePage | Express 版缺失 |
| `GET /api/files/all` | **是** — CommandPalette, AddContextPopover | Express 版缺失 |
| `GET /api/providers/raw` | **是** — ModelSettingsPanel | Express 版缺失 |
| `GET /api/composer/events` | **是** — App.tsx | Express 版缺失 |
| `POST /api/composer/forward` | **是** — VSCode 桥接 | Express 版缺失 |
| `POST/GET /api/remote/sessions` | 否（内部用） | 远程桥接预留 |

**结论：server-node 的路由覆盖度更高，补 2 个 API 即可完全覆盖 server。**

### 3.3 Chat Handler 深度差距（最关键）

server (Express) 有但 server-node 缺失的 14 项关键特性：

| # | 特性 | 影响 | 优先级 |
|---|------|------|--------|
| 1 | `mode: 'plan'` — 只读工具 + read_only sandbox | 前端传 plan 模式时走错逻辑 | P0 |
| 2 | `hooks: hookBus` — PreToolUse/PostToolUse/Stop/UserPromptSubmit | 用户自定义 hook 全部失效 | P0 |
| 3 | Session 并发锁 (`sessions.lock.acquire`) | 同 session 并发 chat 消息交叉 | P0 |
| 4 | `verifyAfterAccept` — accept 后自动 typecheck | accept 后无诊断反馈 | P1 |
| 5 | `renderAutoTriggeredSkills` — 自动匹配 skill | skill 自动触发不工作 | P1 |
| 6 | `toolCtx.checkpoint` — 命令执行前自动 checkpoint | run_command 修改无法自动回滚 | P1 |
| 7 | `toolCtx.backgroundTasks` — 后台执行命令 | Agent 无法后台执行 | P1 |
| 8 | `profileId` 参数 — 前端指定 provider | 无法从 UI 指定特定模型 | P2 |
| 9 | `subagent_followup` SSE 事件 | 前端不知子 Agent 完成后需 follow-up | P2 |
| 10 | Subagent 文本节流 (subagentEventBuffer) | 可能 SSE 洪水 | P2 |
| 11 | Progressive Tool Disclosure (CHAT_ONLY_PROFILE) | plan 模式下 Agent 能调写操作 | P2 |
| 12 | `sandbox/approvalPolicy` 传入 buildMessages | Agent 不感知权限边界 | P2 |
| 13 | `contextWindowOverride` for compaction | compaction 无法感知 contextWindow | P3 |
| 14 | `done` SSE 事件格式差异 | server 无 reason，server-node 有 reason | P3 |

### 3.4 Middleware 差距

| Middleware | server | server-node | 风险 |
|-----------|--------|-------------|------|
| Metrics middleware | 真实计数+4xx/5xx | **STUB** (只返回 ok+ts) | 🟡 |
| Auth timing-safe | `crypto.timingSafeEqual` | `Set.has()` | 🟡 Timing attack |
| SSE 连接追踪 | `sseTrack` | 无 | 🟢 |
| 结构化日志 | `requestLogMiddleware` | 只有 `console.log` | 🟢 |

### Phase 4：合并 SSE 通道

**目标**：2 个持久 SSE 通道合并为 1 个。

#### 当前状态

| 通道 | 端点 | 推送的事件 | 消费者 |
|------|------|-----------|--------|
| fsEventClients | `/api/fs/events` | `fs_change`, `skills.changed` | App.tsx, FileTree.tsx |
| composerSubscribers | `/api/composer/events` | `composer.attach`, `workspace.current`, `skills.changed`(冗余) | App.tsx |

#### 变更

**后端（server-node）**：
```
删除 /api/composer/events 端点
删除 composerSubscribers 集合
所有事件统一通过 fsEventClients 推送（改名为 eventClients）
/api/fs/events → /api/events
事件格式统一为 data: {"type": "xxx", ...}
```

**前端**：
```
App.tsx: 删除 /api/composer/events EventSource，只保留 /api/events
FileTree.tsx: /api/fs/events → /api/events
事件路由：
  - type === 'fs_change' → FileTree 刷新
  - type === 'skills.changed' → bumpSkills
  - type === 'composer.attach' → addAttachment
  - type === 'workspace.current' → 更新 workspace
```

**兼容性**：`/api/fs/events` 保留为 `/api/events` 的别名，过渡期两个都响应。

**验证**：
- 文件树刷新正常
- Skill 热更新正常
- VSCode composer 桥接正常
- Workspace 热切换正常

## 五、server-cloud 的处理策略

### 5.1 本次不动 server-cloud

server-cloud 是完全独立的系统，与本地后端零共享：

| 维度 | server-cloud | 本地后端 |
|------|-------------|---------|
| 认证 | JWT + Cookie + 注册/登录 | 静态 Bearer Token |
| 存储 | PostgreSQL + Prisma | JSONL 文件 |
| 文件系统 | SandboxHandle（隔离沙箱） | 直接 fs（宿主机） |
| SSE 协议 | `event: + data:` | `data: {type:...}` |
| 代码共享 | 只用 `@mini/core` `@mini/sandbox` `@mini/storage` | 额外共享 16+ server/src 模块 |

改动本地后端**完全不影响** server-cloud 和 Web 前端。

### 5.2 未来可做的 server-cloud 优化（不在本次范围）

1. **SSE 协议统一**：server-cloud 改用 `data: {type:...}` 格式，或引入适配层
2. **Rate Limiting**：server-cloud 目前无速率限制
3. **共享更多业务逻辑**：把 server/src 下的 SkillStore、Providers 等提取为 `@mini/server-core` 包

## 六、风险与回滚

### 6.1 每步回滚策略

| Phase | 回滚方式 |
|-------|---------|
| Phase 0 | 删除提取的 utils 文件，恢复内联代码 |
| Phase 1 | 删除新增的路由 handler |
| Phase 2 | chat handler 改回原样（git revert） |
| Phase 3 | middleware 改回原样 |
| Phase 4 | 恢复两个 SSE 通道，前端恢复双 EventSource |
| Phase 5 | Electron main.js spawn 路径改回 server |
| Phase 6 | 恢复 server/src/main.ts 路由代码 |

### 6.2 全局回滚

如果任何 Phase 出现无法快速修复的问题：
1. Electron main.js 改回 `server/main.mjs`
2. `pnpm build` 重新打包 server
3. 系统回到迁移前状态

### 6.3 风险点

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| Chat handler 对齐不完全 | 中 | plan 模式/hook/锁等特性有遗漏 | 逐项测试清单 |
| Electron spawn 路径或端口错误 | 低 | 应用无法启动 | 改回即可 |
| server-node 缺少未发现的 server 特有行为 | 中 | 某功能静默失效 | 完整功能回归测试 |
| SSE 通道合并导致事件丢失 | 低 | 文件树/Skill 不刷新 | 保留别名兼容期 |

        │ /api          │ /api + /cloud-api │ /api
        │               │                   │
┌───────▼───────────────▼───────────────────▼───────────┐
│           server-node (bare Node)                      │

│  ┌────────────────────────────────────────────────┐   │
