# 我用 AI 写了一个「用 AI 写代码」的 IDE，还能用微信遥控

> 5 万行代码，从零手搓一个 Cursor 平替，顺便让微信也能指挥它干活。

---

## 起因：用别人的工具，不如自己造一个

作为一个程序员，我每天和 Cursor、Claude Code 这类 AI 编程工具打交道。它们很好用，但我一直有个疑问：**这些工具到底是怎么把 AI 和代码编辑器结合在一起的？**

与其看文档猜原理，不如自己写一个。于是我花了几个月时间，从零开始写了一个叫 **MiniCodeIDE** 的项目——一个完整的 AI 编程 IDE，包括代码编辑器、AI 对话、文件操作、命令执行，甚至可以用微信远程控制它写代码。

整个项目 200 多个源文件，近 4 万行代码，全部手写，没有 fork 任何开源项目。

---

## 它能做什么？

先说结论：你在 Cursor 里能用到的功能，它基本都有。

**代码编辑**：基于 Monaco Editor（VSCode 同款内核），支持语法高亮、代码跳转、自动补全。你在编辑器里打字，AI 会在光标后面显示灰色的续写建议，按 Tab 就采纳。

**AI 对话**：右侧有个聊天面板，你可以跟 AI 说"帮我重构这个函数"、"这个 bug 怎么修"、"给我写个单元测试"。AI 会读取你当前打开的文件、项目结构，然后给出回答。

**三种模式**：
- **Ask 模式**：纯问答，AI 不会动你的文件，适合咨询和讨论
- **Agent 模式**：AI 可以读文件、写代码、跑命令，像一个真正的编程助手
- **Plan 模式**：AI 先列出计划，你确认后再执行，适合复杂任务

**文件操作**：AI 修改代码时，会先弹出一个"待审核"的 diff 视图，你点 Accept 才真正写入文件。这比直接改文件安全多了。

**命令执行**：AI 可以帮你跑 npm install、git commit 这些命令，但敏感操作会先征求你同意。

## AI 是怎么「理解」你的代码的？

这是整个项目最有意思的部分。

你打开一个有几百个文件的项目，跟 AI 说"帮我找到处理用户登录的函数"。AI 不可能把所有文件都读一遍（太贵也太慢），它需要一种聪明的方式来「检索」相关代码。

我实现了一个**三层混合检索系统**，像一个聪明的图书管理员：

**第一层：关键词匹配**（BM25 算法）
就像 Google 搜索一样，先按关键词找到包含"登录"、"login"这些词的文件。速度快，但可能找到不相关的结果。

**第二层：代码符号图**（Symbol Graph）
把代码里的函数、类、变量当成一张关系网。比如 `handleLogin` 调用了 `validatePassword`，这张网就能帮 AI 顺藤摸瓜找到相关代码。

**第三层：语义向量**（Embedding + Vector Store）
把代码转成数学向量，意思相近的代码在向量空间里距离就近。比如"用户认证"和"登录验证"虽然用词不同，但向量很近。

最后用一个叫 **RRF（Reciprocal Rank Fusion）** 的算法把三层结果融合排序——这其实是学术论文里常用的方法，我把它用到了代码检索上。

## 上下文压缩：让 AI 不「忘事」

跟 AI 聊久了，它会忘记最开始的对话内容。这不是 AI 的 bug，而是所有大语言模型都有「上下文窗口」限制——能处理的文本长度是有限的。

我设计了一个**五级上下文压缩系统**：

1. **保留最近 N 轮对话**：最近的对话完整保留
2. **早期对话摘要**：更早的对话压缩成摘要（用另一个小模型来做摘要）
3. **工具输出裁剪**：AI 读了一个 1000 行的文件，只保留相关的 50 行
4. **磁盘溢出**：实在放不下的内容，先存到磁盘，需要时再加载
5. **懒加载**：文件内容只在 AI 真正需要时才读取

这就像你的大脑——你不会记住昨天午饭吃了什么，但你知道「在某个餐厅吃过」，需要的时候能回忆起来。

## 微信遥控：人在外面，IDE 在家干活

这是我觉得最好玩的功能。

想象一个场景：你在外面吃饭，突然想到一个功能要实现。你打开微信，给家里的 IDE 发条消息："帮我写一个用户注册页面"。IDE 收到消息后，AI 开始工作——读文件、写代码、跑测试——最后把结果发回你微信。

**技术实现**：

我写了一个独立的 Node 进程，通过微信的 iLink Bot 协议（微信官方的 Bot API）接收消息，然后转发给 IDE 的后端。整个链路是：

```
微信消息 → iLink Bot 协议 → 桥接进程 → IDE 后端 → AI 处理 → 回复发回微信
```

桥接进程会自动审批 AI 的工具调用（因为你在微信里没法点 Accept/Reject），相当于把你从电脑前解放出来。

开发过程中踩了不少坑，比如 iLink 的消息发送接口要求 `message_state` 必须是数字 `2`，写成字符串 `"FINISH"` 就会静默失败。这种细节在官方文档里找不到，只能靠反复调试。

## 一些有意思的技术细节

**防幻觉机制**：AI 有时候会「瞎编」——明明没有这个函数，它却说有。我加了五层防御：工具输出校验、重复检测、循环检测、上下文验证、结果回检。

**渐进式工具暴露**：AI 每轮对话可以选择使用哪些工具。Ask 模式下 AI 完全不能用工具（纯问答），Plan 模式下只能用 5 个只读工具（读文件、搜索等），Agent 模式才能用全部 27 个工具（包括写文件和跑命令）。这种分层设计比一次性暴露所有工具，每轮对话节省了约 2400 个 token。

**后端冷启动极快**：整个后端用原生 Node.js 写的（没有用 Express/NestJS 这些重框架），官方设计目标是启动时间 50 毫秒以内。Electron 打包后，用户双击应用图标，几乎瞬间就能开始用。

**200+ 文件的项目结构**：用 pnpm monorepo 管理，6 个应用模块（桌面端、Electron、后端、云端、Web、微信桥接）+ 5 个共享包（核心逻辑、索引器、服务端核心、存储、沙箱）。模块之间通过 TypeScript 的类型系统保证接口一致。

## 写在最后

造一个 AI IDE 听起来是个大工程，但拆解开来其实就是几个核心问题：

1. **怎么让 AI 找到相关代码？** → 混合检索
2. **怎么让 AI 记住上下文？** → 压缩系统
3. **怎么让 AI 安全地操作文件？** → 审批流 + 待审核编辑
4. **怎么让 AI 执行多步任务？** → ReAct 循环（思考-行动-观察）
5. **怎么让这一切跑得快？** → 流式传输 + 懒加载 + 原生后端

每个问题单独看都不算太难，难的是把它们组合成一个流畅的产品。
